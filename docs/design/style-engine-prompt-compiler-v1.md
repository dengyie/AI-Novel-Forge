# 写法引擎 Prompt 与规则编译设计 V1

## 1. 文档目标

本文档用于定义 **写法引擎如何把“写法资产”转换成真正可执行的模型约束**。

核心目标不是“再写一段提示词”，而是建立一条稳定链路：

> **写法资产 → 规则编译 → Prompt 注入 → 输出检测 → 修正重写**

它解决的问题是：

* 写法资产有了，但模型不一定听
* 规则很多，但直接塞给模型会变成一锅粥
* 不同任务需要不同粒度的写法约束
* 反AI规则不能只靠一句“请不要AI味”

---

## 2. 设计原则

### 2.1 不直接把 JSON 原样丢给模型

结构化数据适合存储，不适合直接执行。
必须经过“规则编译器”转成模型易理解、易遵守的文本约束块。

### 2.2 不同任务使用不同 Prompt 结构

大纲生成、正文生成、续写、润色、改写，对写法约束的依赖强度不同。
不能一套 Prompt 打天下。

### 2.3 写法约束必须分层

至少拆成：

* 世界/角色/任务基础层
* 写法规则层
* 反AI约束层
* 当前目标层

### 2.4 规则要分硬约束和软约束

不是所有规则都该用“必须”“禁止”。
有些规则更适合“优先”“倾向”“鼓励”。

### 2.5 输出后检测是 Prompt 体系的一部分

Prompt 不是一次性发射，应该形成闭环：

> 生成前限制
> 生成后检查
> 违规后修正

---

## 3. 核心链路总览

```text
写法资产
↓
规则标准化
↓
规则编译器
↓
任务级 Prompt 组装
↓
模型生成
↓
输出检测器
↓
重写编译器
↓
修正结果
```

---

## 4. 输入来源定义

Prompt 编译器的输入不是只有“写法资产”，而是 5 类上下文共同组成。

### 4.1 基础创作上下文

包括：

* 小说基础设定
* 世界观摘要
* 当前卷目标
* 当前章节目标
* 关键人物状态

### 4.2 写法资产

包括：

* 叙事规则
* 人物表达规则
* 语言风格规则
* 节奏规则
* 默认模板说明

### 4.3 反AI规则

包括：

* 禁止型规则
* 风险型规则
* 鼓励型规则

### 4.4 应用绑定信息

包括：

* 绑定范围
* 优先级
* 强度权重
* 继承关系

### 4.5 当前任务信息

包括：

* 任务类型
* 输入文本
* 目标长度
* 输出形式
* 是否允许重写

---

## 5. Prompt 编译体系分层

建议把 Prompt 编译拆成 6 层。

---

## 5.1 世界与任务基础层

这层解决“写什么”。

内容包括：

* 本书背景
* 本段任务
* 当前人物状态
* 当前情节位置

示例：

```text
当前任务是生成第12章正文。
本章目标：主角在夜场蹲守失败后，与大军发生一次带有嘴硬和互损感的短对话，并在结尾形成新的现实落差。
人物状态：主角囊中羞涩但不愿示弱，大军对外嘴损，对主角实际有一定照应。
```

---

## 5.2 写法主规则层

这层解决“怎么写”。

来源于写法资产中的四大块：

* 叙事规则
* 人物表达
* 语言风格
* 节奏控制

这层是主体，建议统一编译成“规则块”。

示例：

```text
写法要求：
1. 使用碎片化现实流叙事，以时间推进为主，不做大段回顾总结。
2. 每个段落优先呈现人物行为，再显出落差，不要先解释心理。
3. 人物情绪通过动作、语气、对话、环境反应体现，不直接说明“他难过”“他愤怒”。
4. 语言保持口语化和粗粝感，允许不完整句和生活杂音。
5. 收尾不要解决问题，应保留未消化的失败感或现实压迫感。
```

---

## 5.3 角色表达校正层

这层专门处理角色口吻、视角和行为方式。

因为很多 AI 输出最大的问题不是文风错，而是**角色说话像同一个人**。

内容包括：

* 视角规则
* 人物嘴硬程度
* 是否允许自省
* 对话风格
* 角色偏见和误读方式

示例：

```text
角色表达要求：
- 主角说话要嘴硬，遇到尴尬先找补，不直接承认自己吃瘪。
- 大军说话更短，更损，但不做长篇分析。
- 当前视角禁止上帝视角评判，只允许跟随主角感知范围。
```

---

## 5.4 反AI约束层

这层不是“风格建议”，而是专门拦截模型坏毛病。

建议分三段：

### A. 禁止项

```text
禁止出现以下问题：
- 直接解释心理，如“他感到”“他意识到”
- 段尾总结主题
- 抒情式升华
- 工整排比式表达
```

### B. 风险提醒

```text
注意避免以下倾向：
- 连续几段只有解释没有动作
- 对话只推进剧情，没有生活噪音
- 每段都过于完整、工整、像标准作文
```

### C. 鼓励项

```text
优先加入以下特征：
- 无意义但真实的小动作
- 现实落差
- 嘴硬补偿
- 与主线无强关系但有生活感的杂质信息
```

---

## 5.5 输出格式层

这层解决“生成成什么样”。

例如：

* 生成章节正文
* 生成同风格片段
* 生成改写结果
* 生成检测报告

正文任务示例：

```text
输出要求：
- 直接输出正文，不解释写法
- 不分条，不加标题
- 保持小说叙事文本格式
- 长度控制在800到1200字
```

---

## 5.6 自检指令层

这层非常重要。
在 Prompt 末尾加一个极短的自检约束，能显著提高稳定性。

示例：

```text
写完后自行检查：
- 是否出现了直接心理解释
- 是否有段尾升华
- 是否每段都有动作或对话支撑
若存在，先修正再输出最终正文。
```

---

## 6. 规则标准化设计

在编译之前，所有写法规则要先被标准化，不然来源不同会很乱。

---

## 6.1 原始来源例子

拆书可能写成：

* “语言偏市井口语”
* “有诗句点缀”
* “人物内心通过对比显出”

用户手动写的可能是：

* “别太文青”
* “多点脏话”
* “别老解释”

这些都不能直接编译。

---

## 6.2 标准化后格式

统一转成内部规则字段，例如：

```json
{
  "language.register": "colloquial",
  "language.roughness": 0.7,
  "language.allow_poetic_insert": true,
  "character.emotion_expression": "behavior_only",
  "anti.forbid_explicit_psychology": true
}
```

---

## 6.3 标准化阶段职责

标准化器负责：

* 统一词汇
* 去歧义
* 归类到固定字段
* 给出默认值
* 补全缺失项

---

## 7. 规则编译器设计

规则编译器的职责是：

> 把结构化规则翻译成“模型最容易执行”的自然语言规则块。

---

## 7.1 编译器输入

```json
{
  "task_type": "chapter_generation",
  "style_profile": {},
  "anti_ai_rules": [],
  "binding_weight": 0.85,
  "target_scope": "chapter"
}
```

---

## 7.2 编译器输出

输出不是单一字符串，而建议拆成：

```json
{
  "global_context_block": "...",
  "style_rule_block": "...",
  "anti_ai_block": "...",
  "output_block": "...",
  "self_check_block": "..."
}
```

前端或调用层再按模板拼装。

---

## 7.3 编译策略

### 硬规则

使用这些词：

* 必须
* 不得
* 禁止
* 只能

适合：

* 禁止心理直说
* 禁止主题升华
* 必须行为化表达

### 软规则

使用这些词：

* 优先
* 倾向
* 尽量
* 可适当

适合：

* 生活杂音
* 句式断裂
* 无效细节
* 环境噪音

### 风险规则

使用这些词：

* 注意避免
* 不要连续出现
* 警惕

适合：

* 对话过度功能化
* 句式过整齐
* 段落太平衡

---

## 8. 不同任务的 Prompt 模板

---

## 8.1 章节正文生成

适用于从章节目标生成全新正文。

### 编译重点

* 写法规则最强
* 反AI规则强
* 输出格式要求明确
* 自检开启

### 推荐结构

```text
[任务上下文]
[当前人物与情节位置]
[写法规则]
[角色表达规则]
[反AI规则]
[输出要求]
[自检要求]
```

---

## 8.2 续写任务

适用于已有正文后继续往下写。

### 编译重点

* 保持前文语气一致
* 限制模型突然升华
* 更强依赖“承接现有节奏”

附加规则建议：

```text
必须延续现有段落气质，不要突然把语言写得更整齐、更完整、更像总结。
不得擅自抬高主题表达，不得把原本碎片化的生活流改写成有明确中心思想的段落。
```

---

## 8.3 润色任务

适用于已有文本优化。

### 编译重点

* 不是单纯变通顺
* 是“更像当前写法”
* 保留原剧情信息

附加规则建议：

```text
润色时不得新增核心剧情信息，不改变人物关系，不改变事件结果。
只调整表达方式，使文本更符合当前写法资产。
```

---

## 8.4 改写任务

适用于把普通文本改成某种写法。

### 编译重点

* 明确“保留什么，替换什么”
* 规则比正文生成更强
* 检测后通常还要二次修正

附加规则建议：

```text
请在不改变事件顺序与事实信息的前提下，将文本改写为指定写法。
重点改写叙述方式、语言表面、情绪表达方式和段落节奏。
```

---

## 8.5 AI味修正任务

适用于检测后自动重写。

### 编译重点

* 只修违规点
* 尽量少伤原文信息
* 必须带违规报告输入

推荐输入结构：

```text
原文：
...

检测到的问题：
1. 出现“他感到”
2. 第三段存在段尾升华
3. 对话过于功能化

修正要求：
- 保留剧情事实不变
- 仅修改违规表达
- 修正后重新输出全文
```

---

## 9. 写法强度与绑定权重

不是所有写法绑定都要一刀切。
建议引入“强度”概念。

---

## 9.1 强度定义

### 低强度 0.3 ~ 0.5

* 主要提供倾向
* 适合大纲阶段
* 不强压语言细节

### 中强度 0.6 ~ 0.8

* 兼顾规则与灵活性
* 适合正文生成

### 高强度 0.8 ~ 1.0

* 强制约束明显
* 适合改写、试写、风格实验

---

## 9.2 编译中的体现

同一条规则，在不同强度下措辞不同。

例如“禁止解释心理”：

### 中强度

```text
尽量不要直接解释人物心理，优先通过动作和对话体现情绪。
```

### 高强度

```text
禁止直接解释人物心理，不得使用“他感到”“他意识到”等表达。
```

---

## 10. 多层绑定时的合并策略

写法可能同时绑定在：

* 全书
* 卷
* 章节
* 角色视角
* 本次任务

必须有合并规则。

---

## 10.1 推荐优先级

```text
本次任务 > 角色视角 > 章节 > 卷 > 全书模板
```

---

## 10.2 合并原则

### 相同类型规则冲突时

高优先级覆盖低优先级。

### 不冲突规则

进行累加。

### 反AI规则

通常只增不减，除非明确关闭。

---

## 10.3 示例

全书绑定：

* 口语化
* 禁止主题升华

章节绑定：

* 当前章节偏压抑
* 节奏放慢

角色绑定：

* 主角嘴硬
* 不做深度自省

最终编译结果：

* 保留全书口语化和禁止升华
* 叠加章节压抑和慢节奏
* 叠加角色嘴硬和低自省

---

## 11. 反AI规则编译设计

反AI规则不能只是“列清单”，要按类型生成不同文案。

---

## 11.1 禁止型规则编译模板

输入：

```json
{
  "name": "禁止解释型心理描写",
  "type": "forbidden",
  "detect_pattern": ["他感到", "他意识到"]
}
```

编译输出：

```text
禁止直接解释人物心理，不得使用“他感到”“他意识到”等句式。人物状态必须通过动作、对话、环境或行为结果体现。
```

---

## 11.2 风险型规则编译模板

输入：

```json
{
  "name": "对话纯功能推进",
  "type": "risk"
}
```

编译输出：

```text
注意避免对话只承担剧情推进功能。对话中应保留人物语气、停顿、绕弯、嘴硬或无效信息，使其更像真实交流。
```

---

## 11.3 鼓励型规则编译模板

输入：

```json
{
  "name": "鼓励现实落差",
  "type": "encourage"
}
```

编译输出：

```text
优先在段落中加入现实落差，让人物的预期与实际结果之间出现轻微或明显偏差，以增强真实感和人物困境。
```

---

## 12. 自检与二次生成机制

建议把生成分成两种模式。

---

## 12.1 单轮模式

适合轻量任务。

```text
生成 → 自检 → 输出
```

### 优点

* 快
* 成本低

### 缺点

* 稳定性一般

---

## 12.2 双轮模式

适合关键章节、风格敏感任务。

```text
第一轮生成
↓
检测器检查
↓
生成修正指令
↓
第二轮重写
↓
输出最终结果
```

### 优点

* 稳定
* 风格更准

### 缺点

* 成本高

建议正文生成默认单轮，AI味修正和重要章节支持双轮。

---

## 13. 检测报告到重写 Prompt 的转换

这是闭环核心。

---

## 13.1 检测报告格式建议

```json
{
  "risk_score": 76,
  "violations": [
    {
      "rule_id": "anti_001",
      "rule_name": "禁止解释型心理描写",
      "text_span": "他感到一阵烦躁",
      "suggestion": "改为动作或语气表现"
    },
    {
      "rule_id": "anti_005",
      "rule_name": "禁止段尾升华",
      "text_span": "生活终究教会了他……",
      "suggestion": "删除总结，落回具体情境"
    }
  ]
}
```

---

## 13.2 重写 Prompt 模板

```text
请根据以下违规信息修正原文。

修正原则：
1. 不改变事件事实
2. 不新增核心剧情
3. 仅修正违规表达
4. 修正后保持原有写法气质

违规问题：
1. “他感到一阵烦躁”属于直接解释心理，请改为行为化表达
2. “生活终究教会了他……”属于段尾升华，请删除总结感，落回现场感

原文如下：
...
```

---

## 14. 失败保护与降级策略

有时模型会不听话，需要兜底。

---

## 14.1 规则过多时

编译器应做裁剪，优先保留：

1. 当前任务关键写法
2. 高优先级禁止项
3. 当前角色规则
4. 重要鼓励项

不要把二三十条规则全塞进去。

---

## 14.2 任务过短时

例如只生成一句对话，不要注入完整大套写法规则。
应自动切成精简版 Prompt。

---

## 14.3 模型能力较弱时

采用“短规则 + 强约束 + 少抽象词”的编译方式。
少说“叙事气质”“氛围控制”，多说：

* 用短句
* 不解释
* 保持口语
* 不要总结

---

## 15. Prompt 模板示例

---

## 15.1 底层循环现实流章节生成模板

```text
你正在创作一段小说正文。

当前任务：
主角在夜里和朋友蹲场失败，表面嘴硬，实际狼狈。请写出他们撤场后的短暂相处，重点表现人物关系、现实困窘和生活碎音。

写法要求：
1. 采用现实流碎片叙事，以当前时段的行为推进，不做概括式回顾。
2. 每段优先写动作、对话或环境，再显出人物处境，不直接解释心理。
3. 人物的情绪只能通过行为、语气、停顿、反应体现。
4. 语言保持口语化、粗粝感和生活杂音，允许不完整句。
5. 收尾不要解决问题，保留一点狼狈、别扭或落空。

角色表达要求：
- 主角嘴硬，不愿承认吃瘪
- 朋友说话短，损，但不是完全无情
- 当前视角只跟随主角能看到和听到的内容

禁止项：
- 禁止“他感到”“他意识到”等直接心理解释
- 禁止段尾升华
- 禁止把场景写成抒情散文
- 禁止工整排比

优先项：
- 加入无意义但真实的小动作
- 加入现实落差
- 加入一句嘴硬找补
- 加入一点不推动主线的生活性杂音

输出要求：
```

## 16.1 编译器模块拆分

建议拆成以下几个服务：

### `styleNormalizer`

负责把拆书文本、手工描述、模板信息标准化为统一字段。

### `styleMerger`

负责多层绑定合并。

### `styleCompiler`

负责把统一字段编译为 Prompt 规则块。

### `antiAiCompiler`

负责把反AI规则编译成禁止/风险/鼓励块。

### `rewriteCompiler`

负责把检测报告编译成修正 Prompt。

## 17. MVP 范围建议

第一阶段只做最重要的 5 件事：

### 1. 单任务 Prompt 编译

支持章节生成。

### 2. 基础规则编译

支持叙事、语言、角色、反AI四类。

### 3. 4 套模板

底层现实流、爽文推进流、悬疑压迫流、情绪拉扯流。

全书、章节、本次任务。

## 18. 成功标准

这套 Prompt 与规则编译设计做成功，不是看 Prompt 写得多花，而是看下面几点：

1. 同一套写法资产能稳定改变不同任务输出
2. 模型出现八股时能被检测并修正
3. 不同层级绑定不会互相打架
4. 模板、拆书、手工写法都能走进同一套编译链路
5. Prompt 结构清晰，可维护，可扩展

# AI Novel Engine Gap 清单（按优先级排序）

更新时间：2026-03-16
适用范围：当前 `AI-Novel-Writing-Assistant-v2` 代码库现状
用途：替代原先偏愿景化的模块蓝图，直接作为后续开发排序依据

---

## 1. 当前阶段判断

当前项目不是“从零开始”，也不是“已经进入完整导演台阶段”。

更准确的判断是：

> 当前处于 `MVP 已跑通 + 少量 V2 特征已提前出现` 的阶段。

当前已经具备的核心闭环：

```text
项目设定
→ 世界观 / 角色基础资料
→ 发展走向 / 结构化大纲
→ 章节生成
→ 审校 / 修复
→ 自动回写摘要 / facts / 角色时间线
→ 批量流水线继续执行
```

当前最主要的问题不是“功能太少”，而是：

> 系统重心仍偏向“文本生产与修补”，还没有真正转成“叙事状态驱动”。

所以后续开发不应继续优先堆更多生成按钮，而应优先补齐：

1. 状态层
2. 规划层
3. 审计层
4. 导演台控制层

---

## 2. 当前已具备能力

这些能力已经存在，不应再被当成当前主 gap 重复投入：

### 2.1 项目与基础生产能力

- 小说项目、章节、角色、世界观、知识库、拆书模块都已经存在
- 支持发展走向、结构化大纲、章节生成、章节修复、批量 pipeline
- 已有项目概览、章节执行、版本历史等基础工作台页面

### 2.2 AI 与运行时能力

- 已接入 Agent Runtime / Creative Hub / LangGraph
- 已支持多模型提供商与模型路由配置
- 已支持任务中心、运行轨迹、审批中断等机制

### 2.3 知识与检索能力

- 已有知识库文档、版本、绑定、索引任务
- 已有 RAG chunk、向量索引、混合检索
- 已有拆书结果并可回灌到创作流程

### 2.4 轻量状态与版本能力

- 已有 `NovelBible`
- 已有 `PlotBeat`
- 已有 `ChapterSummary`
- 已有 `ConsistencyFact`
- 已有 `CharacterTimeline`
- 已有 `NovelSnapshot`
- 已有 `StorylineVersion`
- 已有 `QualityReport`

结论：

> 现在缺的不是“有没有这些模块”，而是“这些模块是否已经成为系统核心骨架”。

---

## 3. P0：必须最先补齐的 gap

P0 的标准只有一个：

> 不补这些，系统就仍然是“增强版 AI 写作工具”，而不是“AI 小说引擎”。

---

### P0-1. Story State Engine 仍未成型

#### 当前已有

- 章节摘要
- facts 抽取
- 角色时间线
- 角色 `currentState / currentGoal`

#### 核心缺口

- 没有“项目级叙事状态”统一模型
- 没有结构化的角色动态状态表
- 没有结构化关系状态
- 没有信息差状态
- 没有伏笔状态机
- 没有未解决冲突池
- 没有下一章可直接消费的 state snapshot

#### 为什么它是 P0

因为长篇系统的核心不是“再写一章”，而是“写完十章还不崩”。
如果状态层仍然只是摘要、facts 和若干文本字段，后面所有规划、审计、导演台都会虚。

#### 完成标准

- 至少有一套项目级 state schema
- 至少覆盖四类结构化状态：
  - 角色状态
  - 关系状态
  - 信息状态
  - 伏笔状态
- 章节完成后能自动回写状态
- 下一章生成时优先消费状态快照，而不是只拼文本上下文

---

### P0-2. Chapter Planner / Replanner 不独立

#### 当前已有

- outline
- structured outline
- plot beats
- storyline version

#### 核心缺口

- 没有独立的 `Book Planner / Arc Planner / Chapter Planner`
- 没有章级结构化计划对象
- 没有“本章目标 / 场景结构 / 信息揭露 / 情绪推进 / 钩子”的稳定 schema
- 没有偏航后的自动重规划入口
- 没有用户插入事件后的后续推演修正

#### 为什么它是 P0

没有独立规划层，章节 pipeline 只能依赖大纲文本和即时上下文拼接，导致：

- 生成可控性不足
- 偏航后缺少修正支点
- 导演台无法真正操作“高层决策”

#### 完成标准

- Book / Arc / Chapter 至少三层规划对象落库
- 每章生成前必须先有章节计划
- 当章节结果偏离时，可触发 replan
- 用户插入关键事件后，可更新后续章节规划

---

### P0-3. Audit 系统仍偏“质量打分”，不是“叙事审计”

#### 当前已有

- `QualityReport`
- 基于 coherence / repetition / pacing / voice / engagement 的综合评分
- review / repair 流程

#### 核心缺口

- 没有独立 continuity audit
- 没有独立 character audit
- 没有独立 plot audit
- 没有独立 theme audit
- 没有 issue 生命周期管理
- 没有“问题证据 -> 修复建议 -> 是否已解决”的标准闭环

#### 为什么它是 P0

评分可以帮助筛掉低质量文本，但不能维护叙事一致性。
小说引擎要解决的是“世界有没有塌”“角色有没有崩”“信息链有没有打架”，不是只看 overall score。

#### 完成标准

- 审计至少拆成三类：continuity / character / plot
- 每个问题都有结构化 issue 记录
- 支持章节级审计结果留档
- 修复后可标记 issue 是否已解决

---

## 4. P1：高优先级 gap

P1 的标准是：

> P0 补完后，系统要从“能写”进入“能控”，这些模块必须跟上。

---

### P1-1. Character Engine 仍缺动态状态层与关系层

#### 当前已有

- 角色基础资料
- 角色演化
- 角色时间线
- 世界观一致性检查

#### 核心缺口

- 没有角色关系表
- 没有关系强度、信任、冲突、依赖等结构化分值
- 没有秘密状态、认知差、欲望优先级、压力值等状态
- 没有角色弧光进度表达

#### 优先级原因

长篇小说一旦进入中段，真正容易崩的往往不是设定，而是人物行为逻辑和关系变化。

#### 完成标准

- 角色动态状态独立于角色静态档案
- 角色关系可结构化维护
- 角色状态可随章节自动更新

---

### P1-2. Director Console 尚未真正落地

#### 当前已有

- Creative Hub
- 会话式创作中枢
- 生产状态、设定状态、回合总结侧边信息

#### 核心缺口

- 没有明确的“导演台”操作入口
- 不能结构化插入事件
- 不能直接修改角色关系与阶段目标
- 不能 override 状态
- 不能从 UI 上明确发起 replanning

#### 优先级原因

当前系统已有 AI 中枢，但用户仍然更像在“和系统聊天”。
目标中的导演台要求用户操作的是：

- 方向
- 节奏
- 人物命运
- 支线开关
- 主题偏置

这和普通聊天面板不是一回事。

#### 完成标准

- 至少有独立导演台页面或明确导演台区块
- 支持插事件、调关系、改角色目标、重规划、覆盖状态

---

### P1-3. Memory 系统仍以 RAG 为主，缺少分层记忆

#### 当前已有

- 知识库
- 文档绑定
- chunk 索引
- 向量检索与混合检索

#### 核心缺口

- 没有 short / medium / long memory 分层
- 没有项目级 memory item 模型
- 没有专门的事件链检索、人物关系检索、伏笔检索
- 没有把“记忆”和“真相状态”分开

#### 优先级原因

RAG 适合找相关，不适合做叙事真相。
不把记忆层独立出来，后续上下文装配会越来越混乱。

#### 完成标准

- 至少定义短期 / 中期 / 长期记忆层
- 记忆写入与检索有独立模型
- 章节上下文装配明确区分：状态、记忆、参考资料

---

### P1-4. 伏笔与 plot thread 管理缺位

#### 当前已有

- plot beats
- storyline versions

#### 核心缺口

- 没有独立 foreshadowing 实体
- 没有 plot thread 实体
- 没有“已埋下 / 待回收 / 已回收 / 回收失败”状态
- 没有支线起止与优先级管理

#### 优先级原因

这部分直接决定中长篇的结构稳定性，是从“能写章节”走向“能控全书”的关键一步。

#### 完成标准

- 伏笔、支线均有结构化实体
- 可在章节后回写状态
- 可被规划层与审计层共同消费

---

## 5. P2：中优先级 gap

P2 的标准是：

> 这些能力会显著提升系统辨识度和工程质量，但不应抢在 P0 / P1 前面。

---

### P2-1. Versioning & Trace 还不完整

#### 当前已有

- novel snapshot
- storyline version
- agent run / step / approval

#### 核心缺口

- 没有 state snapshot
- 没有规划版本链
- 没有用户 override 历史
- 没有分支创作对比
- 没有统一版本比对视图

#### 完成标准

- 章节、规划、状态都能形成版本链
- 能对比两次关键变更
- 用户手动干预有追踪记录

---

### P2-2. 前端仍偏“生产页”，缺少真正的控制台矩阵

#### 当前已有

- 项目概览
- 章节执行
- 质量修复
- 版本历史
- Creative Hub

#### 核心缺口

- 缺独立 Dashboard
- 缺剧情规划面板
- 缺导演台
- 缺审计中心
- 缺关系图谱 / 时间线总览

#### 完成标准

- 至少补齐四个核心页面：
  - 项目总览
  - 剧情规划面板
  - 导演台
  - 审计中心

---

### P2-3. Style & Voice 还没有成为独立控制系统

#### 当前已有

- 基本模型参数
- 风格相关文本字段
- 写作公式模块

#### 核心缺口

- 没有统一的作品文风配置中心
- 没有视角、叙述气质、句式偏好等结构化控制项
- 没有角色语言风格差异控制

#### 完成标准

- 作品级风格配置独立
- 角色级说话风格有独立表达
- 文风约束可被章节 planner 和 writer 共同消费

---

### P2-4. 工程结构仍偏大服务聚合，领域边界不够清晰

#### 当前已有

- 路由层
- 服务层
- 若干按领域分的子目录

#### 核心缺口

- `novel` 域过重
- planner / state / memory / audit / versioning 还没有明确边界
- 不利于后续持续扩展

#### 完成标准

- 至少把以下领域边界独立出来：
  - planner
  - state
  - memory
  - audit
  - versioning

---

## 6. P3：暂不优先，但可以作为后续方向

这些方向有价值，但当前不应抢资源。

### P3-1. 多分支创作

- 需要等版本链、状态快照、导演台先稳定

### P3-2. 多模型复杂路由优化

- 当前已经有基础模型路由
- 在状态层和规划层不稳之前，进一步复杂化收益有限

### P3-3. 风格训练 / 风格融合

- 应建立在稳定文风控制系统之上

### P3-4. 商业节奏优化

- 应等 continuity / plot / character audit 先成型

### P3-5. 协作创作 / 插件系统

- 属于平台化能力，不是当前主线

---

## 7. 当前推荐开发顺序

不要按“哪个页面最酷”排序，按“哪个缺口能改变系统性质”排序。

### 第一批

1. Story State Engine
2. Chapter Planner / Replanner
3. Audit 拆层

### 第二批

4. Character 动态状态与关系层
5. Director Console
6. Memory 分层
7. 伏笔 / 支线管理

### 第三批

8. Versioning & Trace 完整化
9. 前端控制台矩阵
10. Style & Voice 独立化
11. 服务边界重构

---

## 8. 当前不要优先做的事

以下事项在当前阶段容易让项目继续横向膨胀，但不解决核心问题：

- 再加更多“生成一次”的按钮
- 继续把知识库当成状态层替代品
- 优先做花哨关系图，但没有底层关系模型
- 优先做多分支创作，但没有状态快照体系
- 优先做复杂多 Agent 编排，但没有稳定 planner / state / audit 骨架

---

## 9. 最终结论

当前项目已经不是空壳，也不是单纯 demo。
真正的 gap 在于：

> 已有很多外围能力，但还缺“叙事状态、规划、审计、导演控制”这四根主梁。

后续只要按这份清单推进，目标应该非常明确：

```text
从：
文本生成中心

转向：
叙事状态中心
```

这才是当前版本最应该完成的架构转向。

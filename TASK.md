# AI 长篇成书路线图（小白用户导向）

更新时间：2026-03-25  
适用范围：当前 `AI-Novel-Writing-Assistant2` 代码库现状  
目标用户：完全不懂写作、希望通过 AI 引导或全自动规划完成整本小说创作的用户  
当前定位：强辅助型 AI 小说工作台  
目标定位：可稳定带小白完成整本中长篇小说的 AI 导演系统

---

## 1. 产品判断

当前系统已经不是早期 demo，也不只是单一的文本生成器。它已经具备：

- 小说项目、章节、角色、世界观、知识库、拆书等基础资产
- 大纲、结构化大纲、章节生成、批量生成、审校、修复等主链能力
- Runtime、Creative Hub、任务中心、模型路由、写法引擎等生产能力
- 摘要、facts、状态快照、时间线、审校结果等中间资产沉淀

但从“让写作小白写完整本中长篇小说”的目标看，当前系统仍然存在明确边界：

- 更擅长“协助生成和修补章节”，还不擅长“稳定掌控整本书”
- 更接近“高级写作工作台”，还不是“自动导演式长篇引擎”
- 更适合有人盯着逐步使用，还不够适合让小白低认知负担地一路写到完结

本路线图的核心判断是：

> 当前最重要的，不是继续扩展零散写作功能，而是把系统重心从“文本生产”转向“长篇叙事控制”。

---

## 2. 北极星目标

一句灵感输入后，系统应当能够完成以下闭环：

1. 自动理解题材、卖点、长度、叙事模式与节奏类型
2. 自动补全整本书骨架，包括主线、冲突、人物弧光、结局方向
3. 自动拆解为书级、阶段级、章节级计划
4. 按统一的长篇生成主链持续写作，而不是只会生成单章
5. 在写作过程中自动沉淀状态、检测偏航、修当前章或重排后续计划
6. 以尽量少的用户决策负担，持续推进到完整完结

这意味着系统必须优先优化以下能力：

- 低认知负担
- AI 自动判断与默认推荐
- 长程结构控制
- 强记忆和强纠偏
- 全本成书率，而不是局部单章体验

---

## 3. 路线图总览

### P0：先解决“能否写到完结”

P0 的标准只有一个：

> 让一个完全不会写作的用户，输入一句模糊灵感后，系统能自动给出可执行的全书方案，并稳定持续推进写作。

P0 不追求高级编辑体验，不追求复杂专家配置，不追求炫目的多入口；只追求让系统具备真正的整本长篇写作闭环。

### P1：再解决“能否稳定写好”

P1 的目标是从“能写完整本”提升到“写得更像一部结构完整、人物会成长、节奏可控的中长篇小说”。

### P2：最后解决“能否规模化、产品化”

P2 的目标是把已有长篇能力产品化、可视化、模板化，让系统从引擎走向成熟产品。

---

### P0-0 当前创建主流程深化（当前主方向）

**目标**

在不依赖自动导演的前提下，把现有

`创建小说项目 -> 基本信息 -> 故事宏观规划 -> 角色准备 -> 故事主线 -> 结构化大纲`

打造成真正适合长篇网文的主流程，让小白用户先把“这本书卖什么、前 30 章要兑现什么、角色如何持续推进、每一卷怎样升级”确定下来，再进入章节生产。

**核心产品判断**

- 自动导演继续保留，但只作为后续打磨的实验入口，不再作为当前主流程前置。
- 当前优先级不再是“从一句灵感自动起书”，而是“让现有主流程真正能支撑优质长篇网文的持续生产”。
- `基本信息 / 世界切片 / 写法确认` 不应只是配置区，而应成为书级 framing 层。
- `故事主线` 不应继续主要停留在自然语言草稿，而应升级为卷级结构工作台。
- `结构化大纲` 不应只是章级 JSON 编辑器，而应承接卷纲、卷末高潮、卷间重置点与章纲联动。
- `角色准备` 不应只服务开书前建档，而要支持长篇进行中的新增角色、阵营漂移和关系升级。

#### P0-0a 基本信息 / 世界切片 / 写法确认升级

**目标**

让系统在正式规划前，先明确“这本书写给谁看、靠什么吸引追读、前 30 章要给出什么承诺、这本书的世界和写法边界是什么”。

**主要改动**

- 在基本信息中新增书级 framing 字段：
  - `targetAudience`：目标读者
  - `commercialTags`：核心商业标签 / 卖点标签
  - `competingFeel`：竞品感 / 读者熟悉感
  - `first30ChapterPromise`：前 30 章承诺
  - `bookSellingPoint`：本书核心卖点
- 把 `世界切片` 和 `写法确认` 从附属卡片升级为同一层的书级确认包。
- 世界切片除“会用到什么”外，还要明确：
  - 主舞台
  - 本书当前阶段不要越过的边界
  - 前期优先使用的规则和组织
- 写法确认要支持“书级默认写法建议”，但不强制用户在创建阶段理解复杂写法规则。

**最低产出要求**

- 用户创建项目后，系统能明确输出本书的目标读者、核心卖点、前 30 章承诺和世界/写法边界。
- 后续主线、卷纲、章纲都能直接消费这些 framing 字段。

**验收标准**

- 小白用户不理解“商业化开书”术语，也能在 AI 引导下完成书级 framing。
- 后续生成的主线与卷纲，不再只围绕题材，而会显式围绕目标读者、卖点和前 30 章承诺推进。

**当前进度（2026-03-24）**

- 已完成：
  - `Novel` 已新增 `targetAudience`、`bookSellingPoint`、`competingFeel`、`first30ChapterPromise`、`commercialTagsJson`
  - 基本信息页已升级为 `书级定位与基本信息`，可直接填写目标读者、卖点、竞品感、前 30 章承诺和商业标签
  - `书级 framing` 卡片已新增 `帮我填写`，可基于当前书名、一句话概述、类型和文风关键词用 LLM 自动回填 5 个 framing 字段
  - `世界切片` 刷新 prompt 已显式消费书级 framing，不再只围绕题材和故事想法裁剪
  - `写法推荐` 已显式参考目标读者、前 30 章承诺和商业标签给出候选理由
  - `故事宏观规划` 的项目上下文已并入书级 framing 摘要
- 已验证：
  - create -> detail -> update 的小说路由 roundtrip 可保留 5 个 framing 字段
  - `POST /api/novels/framing/suggest` 可返回结构化书级 framing 建议，并已接入创建页和编辑页
  - `shared build`、`server prisma generate/push`、`server build`、`server/client typecheck`、`server/tests/bookFraming.test.js`、`server/tests/routes.test.js` 已通过
- 当前遗留：
  - 仍未新增更强的多轮 AI 立项向导；当前只补了轻量级 `帮我填写` 和现有 `世界切片 / 写法推荐` 的 framing 消费
  - framing 目前已进入故事宏观规划，但还未进一步下沉到 `卷级主线对象` 和 `结构化卷纲` 的显式界面消费

#### P0-0b 角色准备升级为动态角色系统

**目标**

让角色模块从“初始角色建档”升级为“长篇进行中的角色系统控制台”。

**主要改动**

- 在当前角色准备基础上补齐：
  - `新角色候选区`：正文新出现角色的待确认入口
  - `阵营 / 立场状态`：记录角色当前所属阵营和变化
  - `关系阶段`：关系不再只存静态标签，而要能标记当前阶段
  - `卷级角色职责`：记录角色在当前卷承担的功能
  - `出场密度 / 缺席风险`：避免重要角色长时间消失
- 强化“角色演化”与“关系网”联动，而不是只更新单个角色文本字段。
- 支持在长篇推进中对角色进行“升格 / 降级 / 转阵营 / 更换压力角色身份”。

**最低产出要求**

- 角色模块不再只展示静态角色卡，而能显示当前卷内的核心角色结构。
- 新角色在正文中出现后，系统能给出候选确认入口，而不是只留在正文里。

**验收标准**

- 长篇推进 20 章以上后，角色模块仍能帮助用户判断“谁在推进、谁缺席、谁关系变化了”。
- 阵营漂移、关系升级和新角色入场能够被结构化记录，而不是靠人工记忆。

#### P0-0c 故事主线升级为卷级工作台

**目标**

让 `故事主线` 从文本草稿升级为可以直接控制长篇推进的卷级结构对象。

**主要改动**

- 为主线层新增第一类卷级对象，例如：
  - `VolumePlan`
  - `VolumePromise`
  - `VolumeClimax`
  - `EscalationLine`
  - `PayoffLedger`
- 每一卷至少要明确：
  - 本卷主承诺
  - 本卷主要冲突升级方式
  - 本卷主角变化
  - 本卷卷末高潮
  - 本卷结束后给下一卷留下的牵引
- 保留现有版本控制和影响分析，但对象要从“主线文本版本”升级为“卷级结构版本”。

**最低产出要求**

- 主线模块不再只是自然语言草稿，而是至少能看到每一卷的推进职责和兑现目标。
- 修改主线时，系统能明确提示影响的是哪一卷、哪些角色、哪些章节范围。

**验收标准**

- 用户可以不理解专业“卷纲”术语，也能在界面上看懂：这一卷要打什么、收什么、下卷怎么接。
- 主线改动后，影响分析不再停留在模糊提示，而能具体映射到卷和章节。

**当前进度（2026-03-25）**

- 已完成：
  - `VolumePlan`、`VolumeChapterPlan`、`VolumePlanVersion`、`VolumeImpactResult`、`VolumeSyncPreview` 已作为共享卷级对象落地，Prisma 已新增卷级主资产表与迁移脚本。
  - 已新增 `NovelVolumeService` 与 `/novels/:id/volumes*` 路由，支持卷级工作台的获取、保存、生成、旧数据回填、版本草稿、激活、冻结、diff 与影响分析。
  - 旧 `outline`、`structuredOutline` 与 `StoryPlan.level="arc"` 已改为卷级主资产的兼容来源或派生缓存；卷级版本激活时会自动回写兼容数据。
  - `故事主线` 前端已升级为卷级工作台，可直接维护每卷标题、主承诺、升级方式、主角变化、卷末高潮、下卷承接与未兑现事项。
  - 卷级生成已显式消费书级 framing 与 `故事宏观规划 / 约束引擎` 上下文，不再只围绕自然语言主线草稿生成。
  - 版本控制对象已从“主线文本版本”升级为“卷级结构版本”，影响分析可返回受影响卷和章节范围。
- 已验证：
  - `shared build`
  - `server prisma generate`
  - `server/client typecheck`
  - `pnpm lint`
  - `pnpm build`
  - 旧项目首次读取时，可从 `structuredOutline / outline / StoryPlan.arc` 惰性回填卷级工作台。
- 当前遗留：
  - 卷级版本链路、影响分析和章节同步的路由 / 集成测试仍需补齐。
  - `卷级角色职责` 仍未进入角色系统主界面消费，与 `P0-0b` 的耦合还不完整。

#### P0-0d 结构化大纲升级为卷纲 / 章纲联动工作台

**目标**

让 `结构化大纲` 真正承接卷级结构，而不是只做章级 JSON 的可视化编辑和同步。

**主要改动**

- 在结构化大纲里把以下内容单独做成显式工作台：
  - 卷纲
  - 卷末高潮
  - 卷间重置点
  - 本卷未兑现事项
  - 下卷承接钩子
- 让“结构化大纲同步到章节执行”时，除了章级差异，还能显示卷级影响。
- 支持按“整卷”生成 / 重排 / 同步，而不是默认只能全文级或单章级处理。
- 将 `圣经`、`拍点` 从后置质检区前移为结构化规划资产，而不是等开始批量生产后才补。

**最低产出要求**

- 结构化大纲至少要能区分“卷级结构”和“章级任务”。
- 同步预览不仅显示第几章变化，还要显示哪个卷的高潮、重置点或承接钩子被影响。

**验收标准**

- 用户在进入章节执行前，已经能看清：每卷怎么收、下一卷怎么接、哪些兑现点还没安排。
- 批量生成前，卷纲、章纲、拍点、圣经不再处于割裂状态。

**当前进度（2026-03-25）**

- 已完成：
  - `结构化大纲` 前端已升级为卷纲 / 章纲联动工作台，章节现在直接挂在所属卷下维护，支持新增、删除、移动以及批量调整冲突强度、目标字数和任务单。
  - 章节同步已改为以 `VolumeChapterPlan` 为主来源，支持 `preserveContent` / `applyDeletes` 选项，并可在应用前查看同步预览。
  - 同步预览与结构编辑已开始体现卷级视角，不再只停留在第几章变化，而是由卷下章节计划统一驱动。
  - 服务端会从卷级主资产自动派生 `structuredOutline` 兼容缓存，并在保存或激活卷级版本时回写旧链路可消费的数据。
  - `PlannerService` 已开始优先读取激活的卷级对象构造章节规划上下文，不再只把旧 `structuredOutline` 当成唯一结构来源。
- 已验证：
  - `server/client typecheck`
  - `pnpm lint`
  - `pnpm build`
  - 现有小说详情查询已可直接返回卷级工作台数据，并在旧数据场景下完成兼容迁移。
- 当前遗留：
  - `圣经`、`拍点` 仍未前移为卷级结构资产，尚未和卷纲 / 章纲工作台形成统一资产面板。
  - 仍缺少更强的“按整卷生成 / 重排 / 同步”批处理能力，当前以卷下章节编辑和同步预览为主。

### P0-1 自动导演模式 v1

**目标**

让用户不需要理解题材结构、节奏模型、人物弧光或章节规划，也能从一句灵感进入“整本可写”的状态。

**当前状态**

已暂停，不作为当前继续开发方向。现有试做版仅视为探索性实现，不计入 P0 已完成产出。

**主要改动**

- 新增“从一句灵感生成全书方案”的自动导演入口
- 让系统自动推断题材、长度、叙事模式、节奏模型、结局方向
- 当用户信息不足时，优先自动补全，必要时只追问最少问题
- 输出结果必须是可继续消费的结构化结果，而不是只给一段建议文本

**关键接口与类型**

- `BookSpec`
- `ArcPlan`
- `ChapterSeedPlan`
- `EndingDirection`
- `HookStrategy`

**最低产出要求**

- 作品定位
- 主线冲突
- 核心卖点
- 主角路径
- 结局方向
- 分阶段推进思路
- 初始章节种子计划

**验收标准**

- 用户输入一句灵感后，系统可自动产出整本书的初始执行方案
- 输出结果可以直接进入后续书级/章级规划，而不是需要人工再次整理
- 用户不需要手工填写复杂专业字段也能继续往下走

**暂停说明**

- 当前方案整体不满意，暂不继续扩展。
- 当前试做版暴露出的主要问题包括：候选阶段输出过重、易超时、结构约束不够稳、解析失败率偏高。
- 后续若重启本方向，应优先重写候选阶段设计，而不是在现有方案上继续堆补丁。
- 在 `P0-0 当前创建主流程深化` 未稳定前，自动导演不应重新抬回主入口，也不应用来定义当前默认数据流。

**已确认的产品规则**

- 小说创建阶段不应把“选择写法资产”作为前置步骤，也不应要求小白用户在立项时先判断哪套写法最适合这本书。
- 写法资产更适合作为“正文开写前的书写层确认”，应在作品方向、题材、主角路径和整体调性初步明确后，由系统结合 `styleTone`、题材和规划结果自动推荐 2-3 套候选写法资产。
- 用户此时只需要做低认知负担的确认：采用某套推荐写法、暂不采用，或稍后再决定；不要求在小说创建页理解写法规则、反 AI 规则或绑定层级。
- 一旦用户确认某套写法资产，系统再将其应用到书级 / 章级 / 本次生成级链路；同时保留后续局部覆盖能力，但不把这种复杂性前置给创建阶段。

### P0-2 统一整本生成主链

**目标**

把单章 runtime、批量 pipeline、创作中枢统一到同一条核心长篇生成链，避免质量和行为分叉。

**当前状态**

核心后端统一已完成，当前暂停继续扩展。该项目前按“主链后端已收拢，前端观察与后续增强未继续推进”理解。

**主要改动**

- 统一上下文装配逻辑
- 统一写法注入逻辑
- 统一状态沉淀、审计、修正与回写流程
- 把“单章一套、整本一套、对话一套”的双轨或多轨链路收拢成主链

**关键接口与类型**

- 统一运行时上下文载荷
- 统一章节生成结果结构
- 统一后处理结果结构
- 统一回写与续章输入结构

**验收标准**

- 手工逐章生成与批量生成的上下文装配结果一致
- 创作中枢发起的章节写作与章节页发起的章节写作不再出现核心行为差异
- 单章、整本、对话驱动写作都沉淀到同一套长篇资产体系

**当前已产出**

- 批量 pipeline 已不再直接走旧的 `chapterWritingGraph.runPipelineChapter(...)` 主路径，而是统一改为通过 `ChapterRuntimeCoordinator.runPipelineChapter(...)` 执行。
- 单章 runtime 与批量 pipeline 已共享同一套请求校验、上下文装配、writer 生成、style review、audit、运行时包产出与章节状态更新主链。
- 已新增 `chapterRuntimePipeline` 模块，用于承接 pipeline runtime 的重试、审阅、修复与通过判定逻辑。
- `ChapterRuntimeCoordinator` 已完成模块化收敛，并重新压回文件长度阈值以内。
- 当前仓库下，`server typecheck`、`server build`、`tests/chapter-runtime-routes.test.js`、`tests/runtime.test.js`、`tests/routes.test.js` 已通过。

**暂停说明**

- 本项当前先停在“核心后端主链统一”这一层，不继续扩大到更多可视化、额外前端展示或更深的 pipeline 诊断能力。
- 后续如重启，应优先做“统一链路的可观测性与验收面板”，其次再考虑进一步清理旧分支实现。

### P0-3 长程结构控制与 Replan

**目标**

让书级、阶段级、章级规划成为每次生成的必经层，并让偏航后可以自动重规划，而不是只报警不修正。

**当前状态**

第一阶段核心后端能力已落地，当前已完成：

- `StoryPlan` 已支持结构职责与阶段元数据：`planRole`、`phaseLabel`、`mustAdvanceJson`、`mustPreserveJson`、`sourceIssueIdsJson`、`replannedFromPlanId`
- `PlannerService.generateChapterPlan(...)` 已开始统一消费 `BookPlan`、`ArcPlan`、最近 open audit issues、最近 creative decisions 与默认结构提示
- `PlannerService.replan(...)` 已从“只重算当前章”升级为“默认重排后续 1-5 章窗口”，当前手动入口默认重排 3 章
- `GenerationContextAssembler` 与章节运行时面板已能读取并展示章节结构职责、阶段标签与必须推进/保留项
- 手动重规划接口已返回受影响章节范围，章节页会明确提示“本次重规划覆盖了哪些章”

**当前产出**

- 章节规划现在会显式带出本章职责，如 `setup / progress / pressure / turn / payoff / cooldown`
- 章节规划会带出阶段标签，以及“本章必须推进 / 本章必须保留”的结构提示
- 重规划已支持基于目标章节向后连续窗口重排，不再局限于单章
- 重规划结果会返回 `primaryPlan`、`generatedPlans`、`affectedChapterIds`、`affectedChapterOrders`、`windowSize`
- 当前相关验证已通过：`shared build`、`server typecheck`、`client typecheck`、`server build`、`tests/planner-normalization.test.js`、`tests/planner-plan-metadata.test.js`、`tests/routes.test.js`

**主要改动**

- 引入明确的书级、阶段级、章节级规划对象
- 每章都要有功能标签，例如铺垫、推进、爆点、回收、转折
- 让章节结果对后续计划产生反馈
- 当章节偏离主线、节奏、人物弧光或伏笔推进时，自动生成未来若干章的重规划结果

**关键接口与类型**

- `BookPlan`
- `ArcPlan`
- `ChapterPlan`
- `ReplanDecision`

**验收标准**

- 每章生成前必须有对应的 `ChapterPlan`
- 章节结果可触发 replan，而不是只记录问题
- 用户插入关键事件后，系统能自动修正后续若干章计划
- 中段长篇写作时，主线不容易断，节奏不容易漂

### P0-4 强记忆底座 v1

**目标**

把“摘要 + facts + 状态快照”升级为统一的长篇记忆层，让系统后续真正消费结构化叙事状态，而不是主要依赖纯文本回顾。

**主要改动**

- 统一定义长篇状态对象
- 建立章节完成后的标准状态回写流程
- 明确下一章上下文装配时的优先消费顺序
- 让重要信息具备冲突检测与显式继承能力

**关键接口与类型**

- `StoryStateSnapshot`
- `CharacterState`
- `RelationshipState`
- `InformationState`
- `ForeshadowState`
- `OpenConflict`

**默认原则**

- 状态优先于摘要
- 摘要优先于原始长文本
- 结构化状态优先用于控制生成，原文只作为补充参考

**验收标准**

- 连续生成多章后，状态能被稳定回写和读取
- 下一章上下文能优先消费结构化状态
- 设定自撞、人物失忆、关系回退等问题有显式检测入口

### P0-5 叙事审计闭环 v1

**目标**

把审计从综合打分升级为可驱动行为的“叙事问题流”，让审计结果能直接决定修当前章还是重排后续计划。

**主要改动**

- 将审计拆为 `continuity / character / plot` 三类问题流
- 每个问题都必须有证据、建议动作、处理状态
- 审计要能驱动两种结果：修当前章，或 replan 后续章节
- 审计结果要沉淀为可追踪资产，而不是仅停留在一次性结果

**关键接口与类型**

- `NarrativeIssue`
- `IssueEvidence`
- `FixAction`
- `ResolutionState`

**验收标准**

- 审计结果不再只是总体分数或模糊建议
- 连续多章写作后，可追踪哪些问题已修复、哪些问题仍影响后续
- 章节偏航时，审计能明确建议“修文”还是“重规划”

## 6. P0 默认数据流

P0 的默认数据流不应再模糊，统一按以下主链执行：

1. 用户创建小说项目，先完成最小基本信息
2. 系统先整理书级 framing：目标读者、核心卖点、前 30 章承诺、商业标签、世界边界
3. 系统生成并确认 `世界切片 + 书级写法建议`
4. 系统完成 `故事宏观规划 / 约束引擎`
5. 系统生成并维护动态角色系统，而不是只做初始角色建档
6. 系统生成 `卷级主线对象`，明确每卷承诺、升级、高潮和承接钩子
7. 系统生成 `结构化卷纲 / 章纲`，并与章节执行链同步
8. 章节生成统一读取 `state + plan + style + world slice + character system + memory`
9. 章节写完后执行 `state sync`
10. 章节写完后执行 `narrative audit`
11. 根据审计结果执行 `fix current chapter` 或 `replan next chapters`
12. 刷新下一章上下文，继续整本写作

自动导演后续若继续保留，只应作为以上主链的可选入口之一，而不是当前唯一或默认入口。

如果后续新增能力无法自然接入这条主链，就不应挤进 P0。

## 8. P1 摘要路线

P1 的进入条件是：P0 已经让系统具备“整本能稳定跑起来”的能力。

P1 重点解决“能不能写得更像一部完整小说”，主要方向包括：

- 角色弧光与关系演化引擎
- 节奏与篇幅控制系统
- 整本文风稳定器
- 结构化记忆升级
- 长篇质量趋势评分体系

P1 的结果应当是：

- 人物不只存在设定，还会发展
- 关系不只存在标签，还会演化
- 中段不容易注水
- 全书文风更稳定
- 系统能判断“是不是越写越虚”

## 9. P2 摘要路线

P2 的进入条件是：系统已经具备 P0 的整本闭环和 P1 的基本质量控制。

P2 重点解决“能不能成熟产品化”，主要方向包括：

- 面向小白的全流程作品工厂
- 多模式创作策略
- 题材与写法模板化
- 可视化长篇控制台
- 完结前全书巡检与修复计划

P2 的结果应当是：

- 系统从引擎变成产品
- 用户不需要理解复杂术语也能使用核心能力
- 长篇创作状态能够被可视化、可解释、可追踪

## 10. 当前不优先做的事

以下事项在当前阶段容易消耗资源，但不能显著提升“小白整本成书率”，因此不应抢在 P0 前：

- 在 `P0-0` 稳定前，把自动导演重新抬回主入口
- 复杂专家编辑器
- 低价值界面重构
- 只优化单章、不优化整本稳定性的局部体验
- 高自由度但高认知负担的专家配置项
- 与长篇主链弱相关的多入口花式生成按钮
- 没有底层结构模型支撑的炫目可视化

判断优先级时统一使用一句标准：

> 这个能力，能不能显著提升一个完全不会写作的人把整本小说写到完结的成功率？

如果答案不是“能明显提升”，优先级就不该高于 P0 主线。

## 13. 最终结论

当前版本最重要的架构转向，不是继续堆叠更多写作工具，而是完成下面这条身份升级：

```text
从：
强辅助型 AI 小说工作台

转向：
可稳定带小白完成整本中长篇小说的 AI 导演系统
```

`TASK.md` 的存在意义，就是确保后续每一轮开发都围绕这个目标推进，而不是再次回到“局部功能很多、整本闭环不足”的老路上。

## 附录 A. 写法引擎细化优化计划（后续优化项）

本节属于后续质量深化规划，不与当前 `P0` 主线抢优先级。  
它解决的不是“能不能写”，而是“写法资产能不能更细、更稳、更可控地服务仿写、迁移和整书生成”。

### A.1 当前问题判断

当前写法引擎已经具备基础能力：
- 可以从文本提取写法资产
- 可以保存原文样本
- 可以绑定反 AI 规则
- 已经开始支持“提取特征 -> 资产保存 -> 编辑页启用/停用”

但从“可精细控制的写法系统”目标看，仍然明显不够细化：
- 提取结果经常停留在摘要层，缺少足够多、足够稳定的特征条目
- 特征粒度不统一，有时是抽象总结，有时是局部语言现象，难以混用
- 特征的“仿写价值”和“原文指纹风险”还没有被稳定拆开
- 用户虽然能勾选，但还看不到更细的强度、分组、用途和作用范围
- 写法资产目前更像单份规则集，还不是“可编译的特征池”
- 章节生成、整书绑定、试写、检测、重写，还没有完全共享同一套特征级解释
- 当 AI 没提取出特征时，用户会直接失去可操作入口，恢复链路不够强

### A.2 后续目标

写法引擎后续应升级为：

> 从参考文本中提取完整写法特征池，允许用户按条目精细控制，再将这些选择稳定编译到书级、章级和任务级写作链路中的写法编排系统。

最终要达到的体验不是“抽出一段写法总结”，而是：
- 用户能看到这段文本到底被拆成了哪些写法特征
- 用户能决定哪些启用，哪些弱化，哪些停用
- 用户能理解每项特征为什么存在、证据是什么、风险是什么
- 系统能把这些特征稳定转成生成规则，而不是只保留自然语言摘要

### A.3 产品原则

- 提取时尽量全量，不主动忽略可能有价值的特征
- 展示时全部可见，不把关键特征藏在 AI 自动判断里
- 应用时精细控制，让用户决定启用、弱化、停用
- 初学者优先，默认给出低认知负担的推荐状态，但不能剥夺手动控制
- AI-first，所有特征识别、归类、合并、去重、风险判断都应优先由 AI 结构化完成
- 不用关键词表、硬编码规则去模拟“写法识别”
- 原文样本、提取结果、用户选择、最终编译结果要完整可追溯

### A.4 写法特征体系重构

后续需要先把“写法特征”定义清楚，形成稳定的特征分类体系。建议至少拆为以下一级类目：
- 叙事特征：视角、时序、推进方式、信息释放方式、收尾方式、留白方式
- 语言特征：句长、口语度、粗粝度、句法重复、脏话/黑话密度、比喻密度
- 对话特征：绕弯、顶嘴、嘴硬、废话、攻击性、信息密度、潜台词强度
- 节奏特征：段落密度、快慢切换、爆点间距、碎片化程度、动作/解释比例
- 情绪特征：压迫感、冷幽默、自嘲、麻木感、羞耻感、兴奋感、反讽感
- 意象特征：高频意象、环境噪音、物件偏好、城市/地域气味、感官偏向
- 角色表达特征：自我反应方式、防御机制、姿态、价值判断、脸面意识
- 指纹特征：强地域标签、口头禅、标志句式、专属意象、专属转折、强原文味表达

每个特征条目后续应至少具备这些字段：
- `label`：特征名称
- `description`：该特征具体描述
- `evidence`：原文证据片段
- `group`：所属分组
- `importance`：整体重要度
- `imitationValue`：对仿写的价值
- `transferability`：对迁移生成的价值
- `fingerprintRisk`：原文指纹风险
- `enabled`：是否启用
- `strength`：作用强度
- `scope`：适用范围
- `keepRulePatch`：完全启用时编译出的规则片段
- `weakenRulePatch`：弱化启用时编译出的规则片段

### A.5 提取链路细化

文本提取后续不应是一轮 AI 输出就直接结束，而应拆成更稳的多阶段链路：

1. 全量初提取  
AI 从文本中尽量多地列出全部写法特征，重点保证覆盖率。

2. 结构化归一化  
把不同表述方式统一成稳定字段，确保前端能直接展示。

3. 去重与合并  
把重复、重叠、上下位关系混乱的特征进行合并，但不应丢失有效信息。

4. 风险标注  
把“仿写价值高”和“原文味太重”分别标出来，而不是混为一个判断。

5. 编译预检查  
确认每个条目都有可执行规则片段，避免生成后只能看不能用。

6. 失败回退重提取  
当特征数过少、字段不完整、规则片段缺失时，自动触发二次提取或修复式提取。

### A.6 编辑页能力细化

写法编辑页后续不应只是“有无勾选”，而应成为一个真正的特征控制台。建议增加：
- 固定展示“提取特征启用”区域，不因空结果而完全消失
- 分组展示全部特征
- 支持搜索、筛选、按组折叠
- 支持“全部启用 / 全部停用 / 仅启用高仿写价值 / 仅启用可迁移特征”
- 每条特征支持三态：启用、弱化、停用
- 每条特征支持强度调节，而不是只有 checkbox
- 每条特征清楚显示证据片段
- 每条特征显示“仿写价值 / 迁移价值 / 指纹风险”
- 展示当前启用后对规则集的即时影响
- 当没有特征时，明确展示原因和“重新提取特征”入口

### A.7 写法资产形态升级

后续要把写法资产从“单份 JSON 规则集”升级为“两层结构”：

第一层：特征池  
- 保存完整提取结果
- 保存原文样本
- 保存 AI 分析摘要
- 保存用户逐项选择结果

第二层：编译结果  
- 根据当前启用/弱化/停用状态，编译出可直接进入生成链路的规则集
- 编译结果用于章节生成、整书生成、试写、检测、重写

这样能避免两个问题：
- 以后想改选择时，不用重新从零提取
- 同一份原文可以派生出多套不同用途的写法资产

### A.8 多用途写法版本

后续应允许同一个特征池派生多种用途版本，而不是只有一份资产：
- `高保真仿写版`：更多保留指纹特征，适合临摹、试写、风格贴近
- `平衡生成版`：保留核心风格但弱化高风险指纹，适合常规章节生成
- `整书迁移版`：保留骨架、节奏、叙事和语言气质，尽量减少原文指纹
- `任务覆盖版`：仅对某次任务临时加强少量特征，不污染整书默认写法

### A.9 作用范围细化

后续每个写法资产或特征集合都需要明确作用范围：
- 书级：全书默认写法基调
- 章级：某章节的局部写法偏置
- 任务级：某次生成、改写、补写时的临时强化

写法链路的理想状态是：
- 书级决定整体基调
- 章级决定局部变化
- 任务级决定本次写作动作
- 编译器按优先级稳定合并，而不是简单覆盖

### A.10 与反 AI 规则的关系

后续需要进一步理清“写法特征”和“反 AI 规则”的关系：
- 写法特征负责“我要怎么写”
- 反 AI 规则负责“我不能怎么写”
- 两者应并列展示，不应混成一个系统
- 两者都应进入最终编译链路
- 当某个写法特征和某条反 AI 规则冲突时，应给出显式提示，而不是静默覆盖

### A.11 试写、检测、重写联动

后续写法引擎不能只在“保存资产”时生效，还要打通：
- 试写预览：切换特征启用状态后立即看到风格变化
- 风格检测：指出当前文本缺少了哪些已启用特征
- 风格重写：根据启用特征自动提出修正方向
- 风格对齐评分：判断文本与当前写法资产的吻合度

理想状态下，用户调整一项特征后，系统应能立即回答：
- 这项特征会怎么影响输出
- 当前文本有没有体现
- 如果没有，应如何改写

### A.12 数据与追踪

后续建议补齐以下持久化能力：
- 原文样本版本历史
- 提取草稿版本历史
- 用户特征选择变更历史
- 编译结果快照
- 失败提取记录
- 二次提取记录
- 试写结果与所用特征快照绑定

这样后续才能支持：
- 回滚到旧版本写法
- 比较不同提取结果
- 分析哪种特征组合最稳定

### A.13 验收指标

写法引擎后续优化不应只看“有没有页面”，而应有明确指标：
- 长文本提取后，特征数不能长期为 0
- 中长文本应能稳定提取出足够数量的条目
- 大部分特征都能生成有效规则片段
- 用户能在编辑页明确看到并操作全部特征
- 启用/停用/弱化操作会稳定影响试写输出
- 整书生成使用写法资产时，风格一致性明显提升
- 高保真仿写版与整书迁移版的输出差异应足够明显

### A.14 建议迭代顺序

后续建议按下面顺序推进：

1. 提取可靠性  
先解决“提不出来、提不稳、提了不可用”。

2. 编辑页显式控制  
把特征全部展示、可搜索、可分组、可三态控制。

3. 编译器升级  
让启用、弱化、停用真正影响最终规则集和生成链路。

4. 多版本写法资产  
从同一特征池派生仿写版、生成版、迁移版。

5. 试写/检测/重写联动  
把写法资产从静态配置升级成可反馈系统。

6. 书级 / 章级 / 任务级统一合并  
完成真正的多层写法控制系统。

### A.15 完成标准

当以下条件都成立时，才可以认为“写法引擎细化优化”阶段基本完成：
- 任意文本提取型写法资产都能稳定看到可操作的特征池
- 用户可以逐条决定启用、弱化、停用
- 特征选择能稳定影响规则编译和试写结果
- 同一原文可以派生多份不同用途的写法资产
- 系统能明确区分“可迁移写法”和“原文指纹”
- 写法资产真正成为整书写作链路中的稳定控制层，而不是附属说明文本

# AI 长篇成书路线图（小白用户导向）

更新时间：2026-03-28  
适用范围：当前 `AI-Novel-Writing-Assistant2` 代码库现状  
目标用户：完全不懂写作、希望通过 AI 引导或全自动规划完成整本小说创作的用户  
当前定位：强辅助型 AI 小说工作台  
目标定位：可稳定带小白完成整本中长篇小说的 AI 导演系统

---

## 1. 产品判断

当前系统已经不是早期 demo，也不只是单一的文本生成器。它已经具备：

- 小说项目、章节、角色、世界观、知识库、拆书等基础资产
- 大纲、结构化大纲、章节生成、批量生成、审校、修复等主链能力
- Runtime、Creative Hub、任务中心、模型路由、写法引擎等生产能力
- 摘要、facts、状态快照、时间线、审校结果等中间资产沉淀

但从“让写作小白写完整本中长篇小说”的目标看，当前系统仍然存在明确边界：

- 更擅长“协助生成和修补章节”，还不擅长“稳定掌控整本书”
- 更接近“高级写作工作台”，还不是“自动导演式长篇引擎”
- 更适合有人盯着逐步使用，还不够适合让小白低认知负担地一路写到完结

本路线图的核心判断是：

> 当前最重要的，不是继续扩展零散写作功能，而是把系统重心从“文本生产”转向“长篇叙事控制”。

---

## 2. 北极星目标

一句灵感输入后，系统应当能够完成以下闭环：

1. 自动理解题材、卖点、长度、叙事模式与节奏类型
2. 自动补全整本书骨架，包括主线、冲突、人物弧光、结局方向
3. 自动拆解为书级、阶段级、章节级计划
4. 按统一的长篇生成主链持续写作，而不是只会生成单章
5. 在写作过程中自动沉淀状态、检测偏航、修当前章或重排后续计划
6. 以尽量少的用户决策负担，持续推进到完整完结

这意味着系统必须优先优化以下能力：

- 低认知负担
- AI 自动判断与默认推荐
- 长程结构控制
- 强记忆和强纠偏
- 全本成书率，而不是局部单章体验

---

## 3. 路线图总览

### P0：先解决“能否写到完结”

P0 的标准只有一个：

> 让一个完全不会写作的用户，输入一句模糊灵感后，系统能自动给出可执行的全书方案，并稳定持续推进写作。

P0 不追求高级编辑体验，不追求复杂专家配置，不追求炫目的多入口；只追求让系统具备真正的整本长篇写作闭环。

### P1：再解决“能否稳定写好”

P1 的目标是从“能写完整本”提升到“写得更像一部结构完整、人物会成长、节奏可控的中长篇小说”。

### P2：最后解决“能否规模化、产品化”

P2 的目标是把已有长篇能力产品化、可视化、模板化，让系统从引擎走向成熟产品。

---

## 4. P0 工程计划

P0 是当前唯一主线。下一轮研发、拆 backlog、排优先级、评估完成度，都应以本节为准。

### 当前开发状态（2026-03-28）

- `P0` 总体状态：`书级 framing`、故事宏观规划 / 约束引擎、动态角色系统、卷战略 / 卷骨架 / 节奏板 / 拆章链路、统一 Prompt Registry、统一章节写作上下文已完成两轮整合，当前主线已从“搭底座”切换为“收默认链路、补状态闭环、补端到端验收”。
- 已从当前待办移除的已完成事项：`P0-0a 基本信息 / 世界切片 / 写法确认升级`、`P0-0b 角色准备升级为动态角色系统`、`P0-0c 故事主线升级为卷级工作台`、`P0-0d 结构化大纲升级为卷纲 / 章纲联动工作台`、`P0-2 统一整本生成主链`。
- `P0-0e` 已完成首轮落地：`VolumeStrategyPlan / critique / beat sheet / adjacent rebalance / readiness` 已入库，`VolumePlan v2` 字段已补齐，卷战略与拆章 prompt 已按 `strategy / skeleton / beatSheet / chapterList / rebalance` 分模块拆开并注册，工作区也已经显式锁定“先卷战略、再节奏板、再拆章”的默认顺序。
- `P0-3` 已从“有结构化 planner”推进到“结构化规划开始进入默认运行链”：章节运行时已能消费 `planRole / phaseLabel / mustAdvance / mustPreserve`，审计高危问题也已能触发 replan 建议或自动重规划。
- `P0-4` 与 `P0-5` 不再是纯设计状态：`StoryStateSnapshot`、`OpenConflict`、`AuditIssue`、`chapterWriteContext / chapterReviewContext / chapterRepairContext` 已进入章节生成、审阅、修复与 runtime package，但还没有完成真正稳定的全链闭环验收。
- 当前剩余重点 1：补齐卷级链路测试、旧数据迁移验证、章节同步和动态角色联调，避免“功能已落地但长链路验收不足”。
- 当前剩余重点 2：把 `书级 framing`、`story macro`、流派模式、动态角色系统、卷级资产继续下沉到章节规划、运行时上下文、审阅和修复消费里。
- 当前剩余重点 3：把 `P0-0e` 从“核心资产已落地”推进到“默认链路稳定”，尤其是 `hard / soft planning`、`uncertainty`、`critique -> rebalance -> 后续消费` 这条线。
- 当前剩余重点 4：把 `P0-4` 与 `P0-5` 从“有底座和局部接线”推进到“默认闭环”，让状态、开放冲突、审计问题和 replan 真正共同驱动续章生成。
- `P0-1 自动导演模式 v1`：已暂停，当前试做版不计入 P0 验收完成。暂停原因是整体方案不满意，且存在失败率高、易超时、候选阶段结构过重、结构化解析不稳定等问题。

### 当前剩余重点（P0 活跃待办）

**目标**

把已经落地的创建主流程资产真正接成可稳定消费的长链路，不再继续为已完成模块保留大段待办说明。

#### 1. 卷级链路补测与联调

- 为卷级版本控制、影响分析、章节同步、旧数据迁移补齐路由 / 集成测试。
- 验证动态角色系统、卷级工作台、Planner、Chapter Runtime 在真实链路中的联调结果。
- 以“旧项目迁移 -> 卷级编辑 -> 同步章节 -> 章节规划 -> 运行时消费”为最小验收链路。

#### 2. 已落地资产的更深消费

- 让 `书级 framing` 不只停留在立项和卷级生成，还要稳定下沉到章节规划、运行时上下文和后续改写链路。
- 让动态角色系统的 `卷级职责 / 阵营漂移 / 关系阶段 / 缺席风险 / 新角色候选` 被后续规划和执行显式消费，而不只停留在角色页展示。
- 让卷级主线、卷纲 / 章纲联动、动态角色系统三者形成同一套默认数据流。

#### 3. 结构化规划资产补齐

- 把 `圣经 / 拍点` 从后置质检区前移为结构化规划资产，并挂到卷级工作台旁边统一维护。
- 补齐按整卷生成 / 重排 / 同步的批处理能力，不再主要依赖逐章手工调整。
- 保持旧 `outline / structuredOutline / StoryPlan` 兼容链路，但新能力默认优先消费卷级资产。

#### 4. P0-0e 卷战略与节奏工作台二期收尾

- 把已落地的 `卷战略建议 -> 卷骨架 -> 卷内节奏板 -> 当前卷拆章` 链路继续做实，而不是重复描述已完成的首轮建设。
- 明确前 `2-3` 卷默认硬规划、后续卷默认软规划，并让 `hard / soft planning` 真正进入后续规划、运行时和 UI 提示消费。
- 补齐卷级 `critique / rebalance / uncertainty` 的后续消费和可见性，避免这些资产只生成不驱动行为。
- 继续保持“没有节奏板就不直接拆章”的强约束，并补齐单卷重生、相邻卷再平衡和章节同步的联动验收。

### P0-0e 卷战略与节奏工作台二期

**目标**

把当前“全书卷骨架生成器”升级为真正的“卷战略与节奏工作台”，让系统先帮小白决定“该怎么分卷、哪些卷必须硬规划、哪些卷只能软规划”，再进入卷骨架、卷内节奏板和当前卷拆章。最终目标不是更早锁死整本书，而是让前 `2-3` 卷可硬规划、后续卷保留弹性，并让系统以 AI-first 方式持续维护跨卷平衡。

**当前状态（2026-03-28）**

- `VolumeStrategyPlan`、`VolumeCritiqueReport`、`VolumeBeatSheet`、`VolumeRebalanceDecision`、`VolumePlanningReadiness` 已经进入工作区文档和前后端链路。
- 卷战略、审稿、卷骨架、节奏板、章节列表、单章细化 prompt 已完成拆模块与注册，不再继续堆在旧的 `volumePlanning.prompts.ts` 中扩写新链路。
- 工作区已显式执行前置依赖：没有卷战略不能生成节奏板，没有节奏板不能直接拆章节列表；当卷骨架或章节列表变更时，会自动清空过期的节奏板和再平衡建议。
- 当前缺口已从“有没有二期资产”转为“这些资产能否稳定驱动后续消费”：`hard / soft planning` 仍缺少更深消费，`uncertainty` 还未形成端到端数据流，`critique / rebalance` 还没有完全变成后续决策的默认输入。

**当前判断**

网文侧问题：

- 当前链路已经有了 `卷内节奏板`，但 beat 设计还偏“结构正确”，离“追读欲强、升级点明确”的网文节奏仍有差距。
- 卷级字段已经比旧版完整，但“这一卷为什么值得追、和前后卷如何形成兑现梯度”仍不够稳定。
- 单卷局部重生虽然会清理旧资产并补再平衡建议，但跨卷平衡仍偏脆弱，后半段规划弹性也不够清晰。

LLM 架构侧问题：

- 当前“生成全书卷骨架”仍属于高熵大任务，任务粒度偏大，容易产出格式正确但语义发虚的结果。
- 当前 schema 约束仍强于语义约束，更多是在保证结构化输出，而不是稳定保证卷与卷的追读力和差异化。
- `critique / rebalance` 已落地，但还没有充分变成后续消费链的默认输入；`uncertainty` 也还未形成完整数据流。
- 当前对“哪些卷是 hard planned、哪些卷是 soft planned”的表达仍不够稳定，也没有把不确定性充分暴露给用户和后续链路。

**主要改动**

1. `卷战略建议`
   系统先给出推荐卷数、前 `2-3` 卷的硬规划建议、后续卷的软规划边界、读者兑现阶梯与升级阶梯，不把“先定死全书卷数”作为小白用户的前置决策。
2. `卷骨架`
   在现有 `title / mainPromise / escalationMode / protagonistChange / climax / nextVolumeHook` 基础上，补出更像网文卷纲的追读字段，并把当前卷与前后卷关系显式化。
3. `卷内节奏板`
   在卷骨架和章节列表之间增加“开卷抓手 -> 第一轮升级 -> 中段反转 -> 高潮前挤压 -> 卷高潮 -> 卷尾钩子”的节奏层；没有 `beat sheet` 时，默认不能直接拆章节列表。
4. `当前卷章节列表 + 单章细化`
   保留“当前卷拆章 + 单章细化 bundle”的现有方向，但默认建立在卷战略、卷骨架和卷内节奏板都已成立的前提上；单卷重生后，系统必须给出相邻卷再平衡建议，而不是只改当前卷。

**关键接口与类型**

- 新增持久化 / 可序列化工件：`VolumeStrategyPlan`、`VolumeStrategyVolume`、`VolumeBeatSheet`、`VolumeBeat`、`VolumeCritiqueReport`、`VolumePlanningReadiness`、`VolumeUncertaintyMarker`、`VolumeRebalanceDecision`。
- `VolumePlan` 从当前骨架升级为 `v2`，补充 `openingHook`、`primaryPressureSource`、`coreSellingPoint`、`midVolumeRisk`、`payoffType`，用于显式表达“这一卷为什么值得追、为什么不会写塌”。
- `VolumeStrategyPlan` 和 `VolumeBeatSheet` 是 `VolumePlan` 的同级规划资产，不把所有层都继续塞回 `VolumePlan` 一个对象。
- Prompt 资产规划必须继续遵守 `server/src/prompting/` 治理：新增 `novel.volume.strategy@v1`、`novel.volume.strategy.critique@v1`、`novel.volume.beat_sheet@v1`、`novel.volume.rebalance.adjacent@v1`，升级 `novel.volume.skeleton@v2`、`novel.volume.chapter_list@v2`，章节细化先保留现有 `chapter_purpose / chapter_boundary / chapter_task_sheet` 路径。
- 下一阶段实现时，不得继续把新 prompt 堆进同一个 `volumePlanning.prompts.ts`；需要按 `strategy / skeleton / beatSheet / chapterList / rebalance` 分模块拆开。

**验收标准**

- 系统能推荐卷数，并明确哪些卷是 `hard planned`、哪些卷是 `soft planned`。
- 每卷都有 `opening hook / pressure source / selling point / payoff type / next hook` 级别的追读设计，而不只是抽象卷摘要。
- 当前卷未生成 `beat sheet` 时，默认不能直接拆当前卷章节列表。
- 单卷重生后，系统能输出相邻卷再平衡建议，而不是只重算当前卷。
- `critique` 能指出：卷间重复、中段塌陷风险、卷尾钩子过弱、主角弧线断裂、后半本过度硬规划。

**已确认的产品规则**

- `卷级工作台` 的核心动作从“生成全书卷骨架”改为“生成卷战略建议 / 确认卷骨架”。
- `卷纲 / 章纲联动工作台` 的核心动作改为“先做当前卷节奏板，再拆当前卷章节列表”。
- 默认前 `2-3` 卷硬规划，后续卷软规划；系统推荐卷数和规划强度，不让小白先拍脑袋决定全书卷数。
- 章节列表默认不能跳过 `卷内节奏板` 直接生成；只有节奏板成立后，才进入拆章和章节细化。

### P0-1 自动导演模式 v1

**目标**

让用户不需要理解题材结构、节奏模型、人物弧光或章节规划，也能从一句灵感进入“整本可写”的状态。

**当前状态**

已暂停，不作为当前继续开发方向。现有试做版仅视为探索性实现，不计入 P0 已完成产出。

**主要改动**

- 新增“从一句灵感生成全书方案”的自动导演入口
- 让系统自动推断题材、长度、叙事模式、节奏模型、结局方向
- 当用户信息不足时，优先自动补全，必要时只追问最少问题
- 输出结果必须是可继续消费的结构化结果，而不是只给一段建议文本

**关键接口与类型**

- `BookSpec`
- `ArcPlan`
- `ChapterSeedPlan`
- `EndingDirection`
- `HookStrategy`

**最低产出要求**

- 作品定位
- 主线冲突
- 核心卖点
- 主角路径
- 结局方向
- 分阶段推进思路
- 初始章节种子计划

**验收标准**

- 用户输入一句灵感后，系统可自动产出整本书的初始执行方案
- 输出结果可以直接进入后续书级/章级规划，而不是需要人工再次整理
- 用户不需要手工填写复杂专业字段也能继续往下走

**暂停说明**

- 当前方案整体不满意，暂不继续扩展。
- 当前试做版暴露出的主要问题包括：候选阶段输出过重、易超时、结构约束不够稳、解析失败率偏高。
- 后续若重启本方向，应优先重写候选阶段设计，而不是在现有方案上继续堆补丁。
- 在 `P0-0 当前创建主流程深化` 未稳定前，自动导演不应重新抬回主入口，也不应用来定义当前默认数据流。

**已确认的产品规则**

- 小说创建阶段不应把“选择写法资产”作为前置步骤，也不应要求小白用户在立项时先判断哪套写法最适合这本书。
- 写法资产更适合作为“正文开写前的书写层确认”，应在作品方向、题材、主角路径和整体调性初步明确后，由系统结合 `styleTone`、题材和规划结果自动推荐 2-3 套候选写法资产。
- 用户此时只需要做低认知负担的确认：采用某套推荐写法、暂不采用，或稍后再决定；不要求在小说创建页理解写法规则、反 AI 规则或绑定层级。
- 一旦用户确认某套写法资产，系统再将其应用到书级 / 章级 / 本次生成级链路；同时保留后续局部覆盖能力，但不把这种复杂性前置给创建阶段。

### P0-3 长程结构控制与 Replan

**目标**

让书级、阶段级、章级规划成为每次生成的必经层，并让偏航后可以自动重规划，而不是只报警不修正。

**当前状态**

第一阶段核心后端能力已落地，当前已完成：

- `StoryPlan` 已支持结构职责与阶段元数据：`planRole`、`phaseLabel`、`mustAdvanceJson`、`mustPreserveJson`、`sourceIssueIdsJson`、`replannedFromPlanId`
- `PlannerService.generateChapterPlan(...)` 已开始统一消费 `BookPlan`、`ArcPlan`、最近 open audit issues、最近 creative decisions 与默认结构提示
- `PlannerService.replan(...)` 已从“只重算当前章”升级为“默认重排后续 1-5 章窗口”，当前手动入口默认重排 3 章
- `GenerationContextAssembler` 与章节运行时面板已能读取并展示章节结构职责、阶段标签与必须推进/保留项
- `NovelCoreReviewService.reviewChapter(...)`、`ChapterRuntimeCoordinator` 已能基于审计高危问题给出 `replanRecommendation`，并在高危场景下自动触发重规划
- 章节写作 / 审阅 / 修复链路已开始共用 `chapterMission / chapterWriteContext / chapterReviewContext / chapterRepairContext`
- 手动重规划接口已返回受影响章节范围，章节页会明确提示“本次重规划覆盖了哪些章”

**当前产出**

- 章节规划现在会显式带出本章职责，如 `setup / progress / pressure / turn / payoff / cooldown`
- 章节规划会带出阶段标签，以及“本章必须推进 / 本章必须保留”的结构提示

- 重规划结果会返回 `primaryPlan`、`generatedPlans`、`affectedChapterIds`、`affectedChapterOrders`、`windowSize`
- 当前相关验证已通过：`shared build`、`server typecheck`、`client typecheck`、`server build`、`tests/planner-normalization.test.js`、`tests/planner-plan-metadata.test.js`、`tests/routes.test.js`

**主要改动**

- 引入明确的书级、阶段级、章节级规划对象
- 每章都要有功能标签，例如铺垫、推进、爆点、回收、转折
- 让章节结果对后续计划产生反馈
- 当章节偏离主线、节奏、人物弧光或伏笔推进时，自动生成未来若干章的重规划结果

**关键接口与类型**

- `BookPlan`
- `ArcPlan`
- `ChapterPlan`
- `ReplanDecision`

**验收标准**

- 每章生成前必须有对应的 `ChapterPlan`
- 章节结果可触发 replan，而不是只记录问题
- 用户插入关键事件后，系统能自动修正后续若干章计划
- 中段长篇写作时，主线不容易断，节奏不容易漂

### P0-4 强记忆底座 v1

**目标**

把“摘要 + facts + 状态快照”升级为统一的长篇记忆层，让系统后续真正消费结构化叙事状态，而不是主要依赖纯文本回顾。

**当前状态**

- `StateService` 已支持 `latest snapshot / chapter snapshot / rebuild`，并提供正式路由读取与重建。
- `syncChapterArtifacts(...)` 与 runtime 侧 `ChapterArtifactSyncService` 已在章节草稿、修复和部分编辑写回后自动触发 `state sync`。
- `StoryStateSnapshot`、`OpenConflict` 已进入 `GenerationContextAssembler`，下一章上下文装配已开始显式消费状态快照、开放冲突和局部结构状态，而不再只依赖摘要文本。
- 当前仍未完成的部分是：状态对象虽然已进入上下文，但还没有形成“状态变化 -> 审计判断 -> 规划调整 -> 下章执行”的稳定默认闭环，也缺少更强的多章趋势验收。

**主要改动**

- 统一定义长篇状态对象
- 建立章节完成后的标准状态回写流程
- 明确下一章上下文装配时的优先消费顺序
- 让重要信息具备冲突检测与显式继承能力

**关键接口与类型**

- `StoryStateSnapshot`
- `CharacterState`
- `RelationshipState`
- `InformationState`
- `ForeshadowState`
- `OpenConflict`

**默认原则**

- 状态优先于摘要
- 摘要优先于原始长文本
- 结构化状态优先用于控制生成，原文只作为补充参考

**验收标准**

- 连续生成多章后，状态能被稳定回写和读取
- 下一章上下文能优先消费结构化状态
- 设定自撞、人物失忆、关系回退等问题有显式检测入口

### P0-5 叙事审计闭环 v1

**目标**

把审计从综合打分升级为可驱动行为的“叙事问题流”，让审计结果能直接决定修当前章还是重排后续计划。

**当前状态**

- 审计已从单一综合打分推进为 `continuity / character / plot / mode_fit` 四类问题流，`AuditReport / AuditIssue` 已可持久化、可列出、可关闭。
- 审计问题已能同步到 `OpenConflict`，runtime package 也已返回 `openIssues / hasBlockingIssues / replanRecommendation`，不再只是一次性显示评分。
- 修复流已经可以带着 `chapterRepairContext` 与指定 `auditIssueIds` 进入 prompt，修复通过后也会尝试自动关闭对应问题。
- 当前仍未完成的部分是：`NarrativeIssue` 还没有完全收口成统一领域模型，问题生命周期面板和跨章追踪仍偏后端化，审计到 replan / repair 的分流规则还需要更稳定的默认策略。

**主要改动**

- 将审计拆为 `continuity / character / plot / mode_fit` 四类问题流
- 每个问题都必须有证据、建议动作、处理状态
- 审计要能驱动两种结果：修当前章，或 replan 后续章节
- 审计结果要沉淀为可追踪资产，而不是仅停留在一次性结果

**关键接口与类型**

- `NarrativeIssue`
- `IssueEvidence`
- `FixAction`
- `ResolutionState`

**验收标准**

- 审计结果不再只是总体分数或模糊建议
- 连续多章写作后，可追踪哪些问题已修复、哪些问题仍影响后续
- 章节偏航时，审计能明确建议“修文”还是“重规划”

## 10. 当前不优先做的事

以下事项在当前阶段容易消耗资源，但不能显著提升“小白整本成书率”，因此不应抢在 P0 前：

- 在 `P0-0` 稳定前，把自动导演重新抬回主入口

- 只优化单章、不优化整本稳定性的局部体验
- 高自由度但高认知负担的专家配置项

- 没有底层结构模型支撑的炫目可视化

> 这个能力，能不能显著提升一个完全不会写作的人把整本小说写到完结的成功率？

## 附录 A. 写法引擎细化优化计划（后续优化项）

本节属于后续质量深化规划，不与当前 `P0` 主线抢优先级。  
它解决的不是“能不能写”，而是“写法资产能不能更细、更稳、更可控地服务仿写、迁移和整书生成”。

### A.1 当前问题判断

当前写法引擎已经具备基础能力：
- 可以从文本提取写法资产
- 可以保存原文样本
- 可以绑定反 AI 规则
- 已经开始支持“提取特征 -> 资产保存 -> 编辑页启用/停用”

但从“可精细控制的写法系统”目标看，仍然明显不够细化：
- 提取结果经常停留在摘要层，缺少足够多、足够稳定的特征条目
- 特征粒度不统一，有时是抽象总结，有时是局部语言现象，难以混用
- 特征的“仿写价值”和“原文指纹风险”还没有被稳定拆开
- 用户虽然能勾选，但还看不到更细的强度、分组、用途和作用范围
- 写法资产目前更像单份规则集，还不是“可编译的特征池”
- 章节生成、整书绑定、试写、检测、重写，还没有完全共享同一套特征级解释
- 当 AI 没提取出特征时，用户会直接失去可操作入口，恢复链路不够强

### A.2 后续目标

写法引擎后续应升级为：

> 从参考文本中提取完整写法特征池，允许用户按条目精细控制，再将这些选择稳定编译到书级、章级和任务级写作链路中的写法编排系统。

最终要达到的体验不是“抽出一段写法总结”，而是：
- 用户能看到这段文本到底被拆成了哪些写法特征
- 用户能决定哪些启用，哪些弱化，哪些停用
- 用户能理解每项特征为什么存在、证据是什么、风险是什么
- 系统能把这些特征稳定转成生成规则，而不是只保留自然语言摘要

### A.3 产品原则

- 提取时尽量全量，不主动忽略可能有价值的特征
- 展示时全部可见，不把关键特征藏在 AI 自动判断里
- 应用时精细控制，让用户决定启用、弱化、停用
- 初学者优先，默认给出低认知负担的推荐状态，但不能剥夺手动控制
- AI-first，所有特征识别、归类、合并、去重、风险判断都应优先由 AI 结构化完成
- 不用关键词表、硬编码规则去模拟“写法识别”
- 原文样本、提取结果、用户选择、最终编译结果要完整可追溯

### A.4 写法特征体系重构

后续需要先把“写法特征”定义清楚，形成稳定的特征分类体系。建议至少拆为以下一级类目：
- 叙事特征：视角、时序、推进方式、信息释放方式、收尾方式、留白方式
- 语言特征：句长、口语度、粗粝度、句法重复、脏话/黑话密度、比喻密度
- 对话特征：绕弯、顶嘴、嘴硬、废话、攻击性、信息密度、潜台词强度
- 节奏特征：段落密度、快慢切换、爆点间距、碎片化程度、动作/解释比例
- 情绪特征：压迫感、冷幽默、自嘲、麻木感、羞耻感、兴奋感、反讽感
- 意象特征：高频意象、环境噪音、物件偏好、城市/地域气味、感官偏向
- 角色表达特征：自我反应方式、防御机制、姿态、价值判断、脸面意识
- 指纹特征：强地域标签、口头禅、标志句式、专属意象、专属转折、强原文味表达

每个特征条目后续应至少具备这些字段：
- `label`：特征名称
- `description`：该特征具体描述
- `evidence`：原文证据片段
- `group`：所属分组
- `importance`：整体重要度
- `imitationValue`：对仿写的价值
- `transferability`：对迁移生成的价值
- `fingerprintRisk`：原文指纹风险
- `enabled`：是否启用
- `strength`：作用强度
- `scope`：适用范围
- `keepRulePatch`：完全启用时编译出的规则片段
- `weakenRulePatch`：弱化启用时编译出的规则片段

### A.5 提取链路细化

文本提取后续不应是一轮 AI 输出就直接结束，而应拆成更稳的多阶段链路：

1. 全量初提取  
AI 从文本中尽量多地列出全部写法特征，重点保证覆盖率。

2. 结构化归一化  
把不同表述方式统一成稳定字段，确保前端能直接展示。

3. 去重与合并  
把重复、重叠、上下位关系混乱的特征进行合并，但不应丢失有效信息。

4. 风险标注  
把“仿写价值高”和“原文味太重”分别标出来，而不是混为一个判断。

5. 编译预检查  
确认每个条目都有可执行规则片段，避免生成后只能看不能用。

6. 失败回退重提取  
当特征数过少、字段不完整、规则片段缺失时，自动触发二次提取或修复式提取。

### A.6 编辑页能力细化

写法编辑页后续不应只是“有无勾选”，而应成为一个真正的特征控制台。建议增加：
- 固定展示“提取特征启用”区域，不因空结果而完全消失
- 分组展示全部特征
- 支持搜索、筛选、按组折叠
- 支持“全部启用 / 全部停用 / 仅启用高仿写价值 / 仅启用可迁移特征”
- 每条特征支持三态：启用、弱化、停用
- 每条特征支持强度调节，而不是只有 checkbox
- 每条特征清楚显示证据片段
- 每条特征显示“仿写价值 / 迁移价值 / 指纹风险”
- 展示当前启用后对规则集的即时影响
- 当没有特征时，明确展示原因和“重新提取特征”入口

### A.7 写法资产形态升级

后续要把写法资产从“单份 JSON 规则集”升级为“两层结构”：

第一层：特征池  
- 保存完整提取结果
- 保存原文样本
- 保存 AI 分析摘要
- 保存用户逐项选择结果

第二层：编译结果  
- 根据当前启用/弱化/停用状态，编译出可直接进入生成链路的规则集
- 编译结果用于章节生成、整书生成、试写、检测、重写

这样能避免两个问题：
- 以后想改选择时，不用重新从零提取
- 同一份原文可以派生出多套不同用途的写法资产

### A.8 多用途写法版本

后续应允许同一个特征池派生多种用途版本，而不是只有一份资产：
- `高保真仿写版`：更多保留指纹特征，适合临摹、试写、风格贴近
- `平衡生成版`：保留核心风格但弱化高风险指纹，适合常规章节生成
- `整书迁移版`：保留骨架、节奏、叙事和语言气质，尽量减少原文指纹
- `任务覆盖版`：仅对某次任务临时加强少量特征，不污染整书默认写法

### A.9 作用范围细化

后续每个写法资产或特征集合都需要明确作用范围：
- 书级：全书默认写法基调
- 章级：某章节的局部写法偏置
- 任务级：某次生成、改写、补写时的临时强化

写法链路的理想状态是：
- 书级决定整体基调
- 章级决定局部变化
- 任务级决定本次写作动作
- 编译器按优先级稳定合并，而不是简单覆盖

### A.10 与反 AI 规则的关系

后续需要进一步理清“写法特征”和“反 AI 规则”的关系：
- 写法特征负责“我要怎么写”
- 反 AI 规则负责“我不能怎么写”
- 两者应并列展示，不应混成一个系统
- 两者都应进入最终编译链路
- 当某个写法特征和某条反 AI 规则冲突时，应给出显式提示，而不是静默覆盖

### A.11 试写、检测、重写联动

后续写法引擎不能只在“保存资产”时生效，还要打通：
- 试写预览：切换特征启用状态后立即看到风格变化
- 风格检测：指出当前文本缺少了哪些已启用特征
- 风格重写：根据启用特征自动提出修正方向
- 风格对齐评分：判断文本与当前写法资产的吻合度

理想状态下，用户调整一项特征后，系统应能立即回答：
- 这项特征会怎么影响输出
- 当前文本有没有体现
- 如果没有，应如何改写

### A.12 数据与追踪

后续建议补齐以下持久化能力：
- 原文样本版本历史
- 提取草稿版本历史
- 用户特征选择变更历史
- 编译结果快照
- 失败提取记录
- 二次提取记录
- 试写结果与所用特征快照绑定

这样后续才能支持：
- 回滚到旧版本写法
- 比较不同提取结果
- 分析哪种特征组合最稳定

### A.13 验收指标

写法引擎后续优化不应只看“有没有页面”，而应有明确指标：
- 长文本提取后，特征数不能长期为 0
- 中长文本应能稳定提取出足够数量的条目
- 大部分特征都能生成有效规则片段
- 用户能在编辑页明确看到并操作全部特征
- 启用/停用/弱化操作会稳定影响试写输出
- 整书生成使用写法资产时，风格一致性明显提升
- 高保真仿写版与整书迁移版的输出差异应足够明显

### A.14 建议迭代顺序

后续建议按下面顺序推进：

1. 提取可靠性  
先解决“提不出来、提不稳、提了不可用”。

2. 编辑页显式控制  
把特征全部展示、可搜索、可分组、可三态控制。

3. 编译器升级  
让启用、弱化、停用真正影响最终规则集和生成链路。

4. 多版本写法资产  
从同一特征池派生仿写版、生成版、迁移版。

5. 试写/检测/重写联动  
把写法资产从静态配置升级成可反馈系统。

6. 书级 / 章级 / 任务级统一合并  
完成真正的多层写法控制系统。

### A.15 完成标准

当以下条件都成立时，才可以认为“写法引擎细化优化”阶段基本完成：
- 任意文本提取型写法资产都能稳定看到可操作的特征池
- 用户可以逐条决定启用、弱化、停用
- 特征选择能稳定影响规则编译和试写结果
- 同一原文可以派生多份不同用途的写法资产
- 系统能明确区分“可迁移写法”和“原文指纹”
- 写法资产真正成为整书写作链路中的稳定控制层，而不是附属说明文本

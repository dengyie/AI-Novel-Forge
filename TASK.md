# AI 长篇成书路线图（小白用户导向）

更新时间：2026-03-22  
适用范围：当前 `AI-Novel-Writing-Assistant2` 代码库现状  
目标用户：完全不懂写作、希望通过 AI 引导或全自动规划完成整本小说创作的用户  
当前定位：强辅助型 AI 小说工作台  
目标定位：可稳定带小白完成整本中长篇小说的 AI 导演系统

---

## 1. 产品判断

当前系统已经不是早期 demo，也不只是单一的文本生成器。它已经具备：

- 小说项目、章节、角色、世界观、知识库、拆书等基础资产
- 大纲、结构化大纲、章节生成、批量生成、审校、修复等主链能力
- Runtime、Creative Hub、任务中心、模型路由、写法引擎等生产能力
- 摘要、facts、状态快照、时间线、审校结果等中间资产沉淀

但从“让写作小白写完整本中长篇小说”的目标看，当前系统仍然存在明确边界：

- 更擅长“协助生成和修补章节”，还不擅长“稳定掌控整本书”
- 更接近“高级写作工作台”，还不是“自动导演式长篇引擎”
- 更适合有人盯着逐步使用，还不够适合让小白低认知负担地一路写到完结

本路线图的核心判断是：

> 当前最重要的，不是继续扩展零散写作功能，而是把系统重心从“文本生产”转向“长篇叙事控制”。

---

## 2. 北极星目标

一句灵感输入后，系统应当能够完成以下闭环：

1. 自动理解题材、卖点、长度、叙事模式与节奏类型
2. 自动补全整本书骨架，包括主线、冲突、人物弧光、结局方向
3. 自动拆解为书级、阶段级、章节级计划
4. 按统一的长篇生成主链持续写作，而不是只会生成单章
5. 在写作过程中自动沉淀状态、检测偏航、修当前章或重排后续计划
6. 以尽量少的用户决策负担，持续推进到完整完结

这意味着系统必须优先优化以下能力：

- 低认知负担
- AI 自动判断与默认推荐
- 长程结构控制
- 强记忆和强纠偏
- 全本成书率，而不是局部单章体验

---

## 3. 路线图总览

### P0：先解决“能否写到完结”

P0 的标准只有一个：

> 让一个完全不会写作的用户，输入一句模糊灵感后，系统能自动给出可执行的全书方案，并稳定持续推进写作。

P0 不追求高级编辑体验，不追求复杂专家配置，不追求炫目的多入口；只追求让系统具备真正的整本长篇写作闭环。

### P1：再解决“能否稳定写好”

P1 的目标是从“能写完整本”提升到“写得更像一部结构完整、人物会成长、节奏可控的中长篇小说”。

### P2：最后解决“能否规模化、产品化”

P2 的目标是把已有长篇能力产品化、可视化、模板化，让系统从引擎走向成熟产品。

---

## 4. P0 工程计划

P0 是当前唯一主线。下一轮研发、拆 backlog、排优先级、评估完成度，都应以本节为准。

### 当前开发状态（2026-03-22）

- `P0` 总体状态：当前阶段已暂停继续扩展，先以状态同步为准，不再在本轮继续推进新开发。
- `P0-1 自动导演模式 v1`：已暂停，当前试做版不计入 P0 验收完成。暂停原因是整体方案不满意，且存在失败率高、易超时、候选阶段结构过重、结构化解析不稳定等问题。
- `P0-2 统一整本生成主链`：核心后端统一已完成，当前暂停继续外扩。已完成的核心产出包括：批量 pipeline 已切换为通过 `ChapterRuntimeCoordinator.runPipelineChapter(...)` 执行；单章 runtime 与批量 pipeline 已统一到同一套请求校验、上下文装配、写法审阅、审计与运行时产物链路；`ChapterRuntimeCoordinator` 已拆分收敛，新增 `chapterRuntimePipeline` 模块承接 pipeline runtime 逻辑；相关 `server typecheck / build / runtime routes tests` 已通过。
- `P0-3 长程结构控制与 Replan`：未开始，因当前阶段在 `P0-2` 后暂停。
- `P0-4 强记忆底座 v1`：未开始，因当前阶段在 `P0-2` 后暂停。
- `P0-5 叙事审计闭环 v1`：未开始，因当前阶段在 `P0-2` 后暂停。

### P0-1 自动导演模式 v1

**目标**

让用户不需要理解题材结构、节奏模型、人物弧光或章节规划，也能从一句灵感进入“整本可写”的状态。

**当前状态**

已暂停，不作为当前继续开发方向。现有试做版仅视为探索性实现，不计入 P0 已完成产出。

**主要改动**

- 新增“从一句灵感生成全书方案”的自动导演入口
- 让系统自动推断题材、长度、叙事模式、节奏模型、结局方向
- 当用户信息不足时，优先自动补全，必要时只追问最少问题
- 输出结果必须是可继续消费的结构化结果，而不是只给一段建议文本

**关键接口与类型**

- `BookSpec`
- `ArcPlan`
- `ChapterSeedPlan`
- `EndingDirection`
- `HookStrategy`

**最低产出要求**

- 作品定位
- 主线冲突
- 核心卖点
- 主角路径
- 结局方向
- 分阶段推进思路
- 初始章节种子计划

**验收标准**

- 用户输入一句灵感后，系统可自动产出整本书的初始执行方案
- 输出结果可以直接进入后续书级/章级规划，而不是需要人工再次整理
- 用户不需要手工填写复杂专业字段也能继续往下走

**暂停说明**

- 当前方案整体不满意，暂不继续扩展。
- 当前试做版暴露出的主要问题包括：候选阶段输出过重、易超时、结构约束不够稳、解析失败率偏高。
- 后续若重启本方向，应优先重写候选阶段设计，而不是在现有方案上继续堆补丁。

### P0-2 统一整本生成主链

**目标**

把单章 runtime、批量 pipeline、创作中枢统一到同一条核心长篇生成链，避免质量和行为分叉。

**当前状态**

核心后端统一已完成，当前暂停继续扩展。该项目前按“主链后端已收拢，前端观察与后续增强未继续推进”理解。

**主要改动**

- 统一上下文装配逻辑
- 统一写法注入逻辑
- 统一状态沉淀、审计、修正与回写流程
- 把“单章一套、整本一套、对话一套”的双轨或多轨链路收拢成主链

**关键接口与类型**

- 统一运行时上下文载荷
- 统一章节生成结果结构
- 统一后处理结果结构
- 统一回写与续章输入结构

**验收标准**

- 手工逐章生成与批量生成的上下文装配结果一致
- 创作中枢发起的章节写作与章节页发起的章节写作不再出现核心行为差异
- 单章、整本、对话驱动写作都沉淀到同一套长篇资产体系

**当前已产出**

- 批量 pipeline 已不再直接走旧的 `chapterWritingGraph.runPipelineChapter(...)` 主路径，而是统一改为通过 `ChapterRuntimeCoordinator.runPipelineChapter(...)` 执行。
- 单章 runtime 与批量 pipeline 已共享同一套请求校验、上下文装配、writer 生成、style review、audit、运行时包产出与章节状态更新主链。
- 已新增 `chapterRuntimePipeline` 模块，用于承接 pipeline runtime 的重试、审阅、修复与通过判定逻辑。
- `ChapterRuntimeCoordinator` 已完成模块化收敛，并重新压回文件长度阈值以内。
- 当前仓库下，`server typecheck`、`server build`、`tests/chapter-runtime-routes.test.js`、`tests/runtime.test.js`、`tests/routes.test.js` 已通过。

**暂停说明**

- 本项当前先停在“核心后端主链统一”这一层，不继续扩大到更多可视化、额外前端展示或更深的 pipeline 诊断能力。
- 后续如重启，应优先做“统一链路的可观测性与验收面板”，其次再考虑进一步清理旧分支实现。

### P0-3 长程结构控制与 Replan

**目标**

让书级、阶段级、章级规划成为每次生成的必经层，并让偏航后可以自动重规划，而不是只报警不修正。

**主要改动**

- 引入明确的书级、阶段级、章节级规划对象
- 每章都要有功能标签，例如铺垫、推进、爆点、回收、转折
- 让章节结果对后续计划产生反馈
- 当章节偏离主线、节奏、人物弧光或伏笔推进时，自动生成未来若干章的重规划结果

**关键接口与类型**

- `BookPlan`
- `ArcPlan`
- `ChapterPlan`
- `ReplanDecision`

**验收标准**

- 每章生成前必须有对应的 `ChapterPlan`
- 章节结果可触发 replan，而不是只记录问题
- 用户插入关键事件后，系统能自动修正后续若干章计划
- 中段长篇写作时，主线不容易断，节奏不容易漂

### P0-4 强记忆底座 v1

**目标**

把“摘要 + facts + 状态快照”升级为统一的长篇记忆层，让系统后续真正消费结构化叙事状态，而不是主要依赖纯文本回顾。

**主要改动**

- 统一定义长篇状态对象
- 建立章节完成后的标准状态回写流程
- 明确下一章上下文装配时的优先消费顺序
- 让重要信息具备冲突检测与显式继承能力

**关键接口与类型**

- `StoryStateSnapshot`
- `CharacterState`
- `RelationshipState`
- `InformationState`
- `ForeshadowState`
- `OpenConflict`

**默认原则**

- 状态优先于摘要
- 摘要优先于原始长文本
- 结构化状态优先用于控制生成，原文只作为补充参考

**验收标准**

- 连续生成多章后，状态能被稳定回写和读取
- 下一章上下文能优先消费结构化状态
- 设定自撞、人物失忆、关系回退等问题有显式检测入口

### P0-5 叙事审计闭环 v1

**目标**

把审计从综合打分升级为可驱动行为的“叙事问题流”，让审计结果能直接决定修当前章还是重排后续计划。

**主要改动**

- 将审计拆为 `continuity / character / plot` 三类问题流
- 每个问题都必须有证据、建议动作、处理状态
- 审计要能驱动两种结果：修当前章，或 replan 后续章节
- 审计结果要沉淀为可追踪资产，而不是仅停留在一次性结果

**关键接口与类型**

- `NarrativeIssue`
- `IssueEvidence`
- `FixAction`
- `ResolutionState`

**验收标准**

- 审计结果不再只是总体分数或模糊建议
- 连续多章写作后，可追踪哪些问题已修复、哪些问题仍影响后续
- 章节偏航时，审计能明确建议“修文”还是“重规划”

## 5. P0 关键接口与类型变化

P0 期间允许新增以下核心类型，并围绕它们收敛后续实现：

- 规划层：`BookSpec`, `BookPlan`, `ArcPlan`, `ChapterSeedPlan`, `ChapterPlan`, `ReplanDecision`
- 状态层：`StoryStateSnapshot`, `CharacterState`, `RelationshipState`, `InformationState`, `ForeshadowState`, `OpenConflict`
- 审计层：`NarrativeIssue`, `IssueEvidence`, `FixAction`, `ResolutionState`
- 运行时层：`chapterPlan`, `storyStateDigest`, `auditIssues`, `recommendedReplanActions`
- 创作入口：自动导演请求，用于“灵感转全书方案”

这些类型的共同要求是：

- 必须可持久化或可稳定序列化
- 必须能被多个模块消费，而不是只服务单点功能
- 必须围绕“整本成书率”设计，而不是围绕某一个页面方便设计

---

## 6. P0 默认数据流

P0 的默认数据流不应再模糊，统一按以下主链执行：

1. 用户输入一句灵感
2. 系统生成 `BookSpec`
3. 系统生成 `BookPlan / ArcPlan / ChapterPlan`
4. 章节生成统一读取 `state + plan + style + memory`
5. 章节写完后执行 `state sync`
6. 章节写完后执行 `narrative audit`
7. 根据审计结果执行 `fix current chapter` 或 `replan next chapters`
8. 刷新下一章上下文，继续整本写作

如果后续新增能力无法自然接入这条主链，就不应挤进 P0。

## 7. P0 测试与验收

### 单元测试

- `BookSpec / ChapterPlan` 的生成、合并、覆盖与更新
- 状态快照的 diff / merge / 冲突检测
- `NarrativeIssue` 生命周期与处理状态流转
- `ReplanDecision` 的触发条件与输出结构

### 集成测试

- 一句灵感生成全书方案
- 连续生成多章后状态被正确沉淀并被下一章消费
- 章节偏航后自动触发 replan
- 批量 pipeline 与单章 runtime 的上下文注入结果一致

### 端到端验收场景

写作小白输入一句题材想法后，系统能够：

- 自动生成 30 章以上的全书方案
- 连续产出前 10 章
- 不出现主角目标丢失
- 不出现核心设定自撞
- 不出现主线突然消失
- 不出现已埋主冲突完全断线

### 文档验收标准

- 新版 `TASK.md` 必须让实施者不依赖旧文档即可直接进入开发
- `P1 / P2` 只保留方向，不与 `P0` 争抢实现优先级
- 后续 backlog 拆分必须以本文件为唯一主依据

## 8. P1 摘要路线

P1 的进入条件是：P0 已经让系统具备“整本能稳定跑起来”的能力。

P1 重点解决“能不能写得更像一部完整小说”，主要方向包括：

- 角色弧光与关系演化引擎
- 节奏与篇幅控制系统
- 整本文风稳定器
- 结构化记忆升级
- 长篇质量趋势评分体系

P1 的结果应当是：

- 人物不只存在设定，还会发展
- 关系不只存在标签，还会演化
- 中段不容易注水
- 全书文风更稳定
- 系统能判断“是不是越写越虚”

---

## 9. P2 摘要路线

P2 的进入条件是：系统已经具备 P0 的整本闭环和 P1 的基本质量控制。

P2 重点解决“能不能成熟产品化”，主要方向包括：

- 面向小白的全流程作品工厂
- 多模式创作策略
- 题材与写法模板化
- 可视化长篇控制台
- 完结前全书巡检与修复计划

P2 的结果应当是：

- 系统从引擎变成产品
- 用户不需要理解复杂术语也能使用核心能力
- 长篇创作状态能够被可视化、可解释、可追踪

---

## 10. 当前不优先做的事

以下事项在当前阶段容易消耗资源，但不能显著提升“小白整本成书率”，因此不应抢在 P0 前：

- 复杂专家编辑器
- 低价值界面重构
- 只优化单章、不优化整本稳定性的局部体验
- 高自由度但高认知负担的专家配置项
- 与长篇主链弱相关的多入口花式生成按钮
- 没有底层结构模型支撑的炫目可视化

判断优先级时统一使用一句标准：

> 这个能力，能不能显著提升一个完全不会写作的人把整本小说写到完结的成功率？

如果答案不是“能明显提升”，优先级就不该高于 P0 主线。

## 11. 执行顺序与里程碑

### 第一阶段：让系统能从灵感走到整本可写

- `P0-1 自动导演模式 v1`
- `P0-2 统一整本生成主链`

完成标志：

- 用户可从一句灵感进入整本方案
- 单章、整本、对话驱动写作不再分叉成不同质量体系

### 第二阶段：让系统能持续控制整本走向

- `P0-3 长程结构控制与 Replan`

完成标志：

- 每章都有明确功能定位
- 偏航后系统可以自动改后续，不只是提示问题

### 第三阶段：让系统能稳定纠偏而不是越写越漂

- `P0-4 强记忆底座 v1`
- `P0-5 叙事审计闭环 v1`

完成标志：

- 状态、记忆、审计开始共同驱动续章生成
- 长篇连续写作时，结构和角色不容易自然失控

### 并行边界

允许并行：

- `P0-1` 与 `P0-2` 的方案设计和接口定义
- `P0-4` 与 `P0-5` 的类型设计

不建议并行：

- 在 `P0-2` 主链统一之前，提前大规模扩展 P0-3 / P0-4 / P0-5 的实现
- 在 `P0` 未稳定前，提前推进 P1 的复杂质量功能

## 12. 默认决策

- `TASK.md` 采用整体替换，不保留旧版双层正文
- 旧路线图通过 Git 历史追溯，不再在正文中并存
- `P0` 用工程落地版表达，不写成汇报型路线图
- 全文只保留一个主叙事：让不会写作的人在 AI 导演下完成整本中长篇小说
- 所有与该目标弱相关的能力，统一降级到 `P1 / P2` 或“不优先”

## 13. 最终结论

当前版本最重要的架构转向，不是继续堆叠更多写作工具，而是完成下面这条身份升级：

```text
从：
强辅助型 AI 小说工作台

转向：
可稳定带小白完成整本中长篇小说的 AI 导演系统
```

`TASK.md` 的存在意义，就是确保后续每一轮开发都围绕这个目标推进，而不是再次回到“局部功能很多、整本闭环不足”的老路上。
